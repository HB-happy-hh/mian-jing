# 美团 AI 面试题解答

## 1. 在线歌单系统数据库设计

**实体关系分析：**
- 歌手（Artist）与专辑（Album）：多对多（一个歌手有多张专辑，一张专辑可多个歌手合作）
- 专辑（Album）与歌曲（Song）：一对多（一张专辑包含多首歌）
- 歌单（Playlist）与歌曲（Song）：多对多（一个歌单有多首歌，一首歌可在多个歌单）
- 用户（User）与歌单（Playlist）：一对多

**表设计：**

```sql
-- 歌手表
artist(id, name, avatar, bio, created_at)

-- 专辑表
album(id, title, cover, release_date, description)

-- 专辑-歌手关联（多对多）
album_artist(album_id, artist_id, role)  -- role: 主唱/合作等

-- 歌曲表
song(id, title, album_id, duration, url, lyric_url, track_no)

-- 歌曲-歌手关联（多对多，处理feat情况）
song_artist(song_id, artist_id, role)

-- 用户表
user(id, username, avatar)

-- 歌单表
playlist(id, user_id, name, cover, description, is_public, created_at)

-- 歌单-歌曲关联（多对多）
playlist_song(playlist_id, song_id, order_no, added_at)
```

**索引策略：**
- `playlist_song` 上对 `playlist_id` 建索引
- `song` 上对 `album_id` 建索引
- 热门歌单/歌曲可加 Redis 缓存

**MySQL 查询优化（结合歌单系统）：**

**1. 索引优化**
- 为高频查询字段建索引：`playlist_song(playlist_id)`、`song(album_id)`、`song_artist(song_id)`
- 联合索引遵循**最左前缀原则**，如 `(playlist_id, order_no)` 支持按歌单排序查询
- 避免在索引列上做函数运算或隐式类型转换，否则索引失效

**索引失效常见场景举例：**

```sql
-- 假设 song 表有索引：idx_title(title)、idx_album_track(album_id, track_no)

-- ❌ 1. 对索引列使用函数
SELECT * FROM song WHERE LEFT(title, 3) = 'Love';
-- ✅ 改为：WHERE title LIKE 'Love%'

-- ❌ 2. 隐式类型转换（title是varchar，传入数字）
SELECT * FROM song WHERE title = 123;
-- MySQL会对title列做隐式转换，索引失效
-- ✅ 改为：WHERE title = '123'

-- ❌ 3. 联合索引违反最左前缀，跳过了第一列
SELECT * FROM song WHERE track_no = 1;
-- idx_album_track(album_id, track_no) 必须先用 album_id
-- ✅ 改为：WHERE album_id = 10 AND track_no = 1

-- ❌ 4. LIKE 以通配符开头
SELECT * FROM song WHERE title LIKE '%Love%';
-- ✅ 改为前缀匹配：WHERE title LIKE 'Love%'

-- ❌ 5. 使用 OR 连接，其中一个字段无索引
SELECT * FROM song WHERE title = 'Love' OR duration = 180;
-- duration 没有索引，导致整个查询全表扫描
-- ✅ 改为 UNION：
--    SELECT * FROM song WHERE title = 'Love'
--    UNION SELECT * FROM song WHERE duration = 180

-- ❌ 6. 对索引列进行运算
SELECT * FROM song WHERE id + 1 = 100;
-- ✅ 改为：WHERE id = 99

-- ❌ 7. IS NOT NULL（在某些情况下索引失效）
SELECT * FROM song WHERE title IS NOT NULL;
-- 当不为NULL的数据量占大多数时，优化器可能放弃索引走全表

-- ❌ 8. 联合索引中间列使用范围查询，后续列失效
SELECT * FROM song WHERE album_id > 10 AND track_no = 1;
-- album_id 用了范围查询，track_no 的索引部分失效
```

**2. EXPLAIN 分析执行计划**
```sql
EXPLAIN SELECT s.title FROM playlist_song ps
JOIN song s ON ps.song_id = s.id
WHERE ps.playlist_id = 100 ORDER BY ps.order_no;
```
重点关注：
- `type`：`ref` / `range` 较好，`ALL`（全表扫描）需优化
- `key`：实际使用的索引
- `rows`：预估扫描行数，越小越好
- `Extra`：出现 `Using filesort` / `Using temporary` 需警惕

**3. 慢查询定位**
```sql
-- 开启慢查询日志
SET GLOBAL slow_query_log = ON;
SET GLOBAL long_query_time = 1;  -- 超过1秒记录
-- 用 mysqldumpslow 或 pt-query-digest 分析日志
```

**4. 分页优化**
```sql
-- 深分页性能差（offset越大越慢）
SELECT * FROM song LIMIT 100000, 10;

-- 优化：用主键/游标分页
SELECT * FROM song WHERE id > 100000 LIMIT 10;
```

**5. 其他常见优化手段**
- **覆盖索引**：查询字段全在索引中，避免回表
- **避免 SELECT \***：只查需要的列
- **小表驱动大表**：JOIN 时用小结果集驱动大表
- **批量写入代替循环单条 INSERT**
- **读写分离**：主库写，从库读，缓解查询压力
- **热点数据上 Redis**：歌单详情、热门歌曲等加缓存层

## 2. JVM 内存区域

| 区域 | 线程共享 | 作用 | 特点 |
|------|----------|------|------|
| **堆（Heap）** | 是 | 存放对象实例、数组 | GC 主要区域，分新生代/老年代；OOM 报 `java.lang.OutOfMemoryError: Java heap space` |
| **虚拟机栈（Stack）** | 否（每线程一个） | 存放栈帧（局部变量表、操作数栈、方法返回地址） | 递归过深抛 `StackOverflowError`；线程私有 |
| **本地方法栈** | 否 | 为 native 方法服务 | 与虚拟机栈类似，HotSpot 合二为一 |
| **方法区（MetaSpace）** | 是 | 存类信息、常量、静态变量、JIT 编译后代码 | JDK8 后改为元空间（MetaSpace），使用本地内存，避免永久代 OOM |
| **程序计数器** | 否 | 记录当前线程执行的字节码行号 | 唯一不会 OOM 的区域；执行 native 方法时值为 undefined |
| **运行时常量池** | 是 | 存放字面量和符号引用 | 属于方法区的一部分 |

## 3. TCP vs UDP 及多人游戏通信选型

**区别对比：**

| 特性 | TCP | UDP |
|------|-----|-----|
| 连接 | 面向连接（三次握手） | 无连接 |
| 可靠性 | 可靠，有重传机制 | 不可靠，可能丢包 |
| 顺序 | 保证顺序 | 不保证顺序 |
| 延迟 | 较高（拥塞控制、重传） | 低 |
| 适用场景 | 文件传输、HTTP | 直播、游戏、DNS |

**实时多人游戏选 UDP，原因：**
- 游戏对实时性要求极高（< 100ms），TCP 重传机制会导致队头阻塞
- 游戏数据时效性强，丢了旧包意义不大，不需要重传旧状态

**UDP 丢包/可靠性问题的解决方案：**
1. **应用层序列号**：每包带序号，接收方检测丢包和乱序
2. **ACK 确认 + 选择性重传**：只重传关键包（如玩家操作），位置/状态用最新包覆盖
3. **FEC（前向纠错）**：发送冗余数据，少量丢包无需重传即可恢复
4. **状态同步而非帧同步**：丢包后用最新状态覆盖，避免对旧包依赖
5. **QUIC 协议**：基于 UDP 实现了多路复用和部分可靠性，Google 已广泛使用

## 4. 缓存穿透

**定义：** 查询一个数据库中**根本不存在**的 key，缓存不命中，每次都打到 DB，DB 也查不到，反复穿透缓存层。

**危害：** 大量恶意请求（如 id=-1）可打垮数据库。

**解决方案：**

1. **缓存空值**：查询结果为空时，仍将 `null` 写入 Redis，设置短 TTL（如 60s）
   - 缺点：可能缓存大量无效 key

2. **布隆过滤器（推荐）**：启动时将所有合法 key 加入布隆过滤器，请求先经过过滤器，不存在的直接拒绝
   - 优点：内存极小、效率高
   - 缺点：有误判率，不支持删除（可用布谷鸟过滤器替代）

3. **接口限流/鉴权**：非法参数直接在网关/业务层拦截

4. **id 合法性校验**：在查询前校验 id 格式、范围，不合法直接返回

## 5. 后端 Token 过多时向量库与 KV 的设计

背景：LLM/AI 应用中 context token 过长，需要外部记忆存储来压缩 Prompt。

**整体架构：**

```
用户对话 → 提取关键信息 → 存储
         ↓                  ↓
    向量库（语义检索）    KV存储（精确检索）
         ↓                  ↓
    召回相关记忆  →  拼接进 Prompt → LLM
```

**KV 存储（Redis）：**
- 存储结构化、精确的会话信息：`user:{id}:session` → 最近 N 轮对话
- 存储用户偏好、基本信息等固定字段
- 按 session_id 或 user_id 精确查询，延迟极低

**向量库（如 Milvus / Qdrant / Pinecone）：**
- 将历史对话/文档 Embedding 后存储
- 每次新对话时，将当前 query Embedding，召回最相关的历史记忆（Top-K）
- 解决长期记忆问题，支持语义检索

**混合策略：**
- 短期记忆（近 5 轮）→ KV 直接放入 context
- 长期记忆（历史会话）→ 向量库语义检索，Top-K 召回后注入 Prompt
- Token 压缩：对历史对话做摘要再存储，减少 Embedding 和召回的 token 量

## 6. 在资源/时间/技术有限情况下的经历

> 此题为行为面试题，用 **STAR 法则**回答，以下为参考框架：

**S（Situation）背景：**
在某次项目迭代中，临近上线前两天，后端核心开发人员请假，同时需求追加了新功能，时间、人力双重压缩。

**T（Task）任务：**
在保证主流程稳定上线的前提下，尽可能覆盖新增需求。

**A（Action）行动：**
- 快速梳理需求优先级，将功能拆分为"必须上线"和"可延期"两类（MVP 思维）
- 砍掉非核心功能，使用成熟中间件代替自研（如用现成 OSS SDK 代替自写上传逻辑）
- 与产品沟通，达成一致的范围边界，避免需求蔓延
- 对剩余工作做合理分工，聚焦在关键路径上
- 提前做好降级预案，新功能加开关，出问题可快速关闭

**R（Result）结果：**
主流程如期上线，非核心功能顺延至下一个迭代，线上无重大故障，后续补齐了延期功能。

**核心体现：** 在有限条件下做出最优决策的工程判断力，而非硬堆资源。

## 7. Java OOM 排查：如何区分堆问题还是栈问题

**第一步：看报错信息**

| 错误信息 | 问题所在 |
|----------|----------|
| `OutOfMemoryError: Java heap space` | 堆内存不足 |
| `OutOfMemoryError: Metaspace` | 方法区/元空间溢出（类加载过多） |
| `OutOfMemoryError: GC overhead limit exceeded` | GC 占用时间过多，堆问题 |
| `StackOverflowError` | 栈溢出（递归过深） |
| `OutOfMemoryError: unable to create native thread` | 线程过多，栈空间耗尽 |

**第二步：堆问题排查**

```bash
# 生成堆转储文件
jmap -dump:format=b,file=heap.hprof <pid>
# 或 JVM 启动参数（推荐，OOM时自动触发）
-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/path/heap.hprof

# 查看 GC 实时情况
jstat -gcutil <pid> 1000   # 每秒打印 GC 统计

# 用 MAT（Memory Analyzer Tool）或 JProfiler 分析 hprof 文件
# 重点看：大对象、内存泄漏的 GC Root 引用链
```

**第三步：栈问题排查**

```bash
# 查看所有线程栈快照
jstack <pid> > thread.txt
# 搜索 BLOCKED / WAITING 状态线程，看是否有大量线程积压或递归调用链
```

**第四步：常见原因与解法**

| 原因 | 解法 |
|------|------|
| 内存泄漏（静态集合持有对象引用） | MAT 分析 GC Root，找泄漏点 |
| 大数据量一次性加载到内存 | 分页查询 / 流式处理 |
| 递归无终止条件 | 检查代码逻辑，改为迭代实现 |
| 线程池配置过大 | 限制最大线程数，合理设置队列 |
| MetaSpace 溢出（动态代理/反射/热部署） | 增大 `-XX:MaxMetaspaceSize` 或检查类加载器泄漏 |
| 直接内存溢出（NIO ByteBuffer） | 检查 `DirectByteBuffer` 使用，限制 `-XX:MaxDirectMemorySize` |
