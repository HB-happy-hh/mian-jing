# 1. 前后端分离，你是怎么处理跨域的

开发的时候，前端 Vue 项目会在 vue.config.js 里配置代理，把 /api 开头的请求转发到后端的地址，这样浏览器就以为是同源请求了。上线之后，一般会用 Nginx 反向代理，把前端和后端的请求都指向同一个域名，彻底解决跨域问题。

## 2. 你说用 JWT 做登录认证，JWT 存在哪里？前端 localStorage 还是 cookie？

JWT 通常存在前端的 localStorage 中。用户登录成功后，后端生成 JWT 返回给前端，前端把它存在 localStorage，之后每次请求时，在请求头的 Authorization 字段里带上这个 token 传给后端验证。

## 3. JWT 能不能被篡改？怎么防篡改？

JWT能被篡改，但改了就会失效。它的防篡改核心是签名机制，后端用密钥对header和payload生成签名，前端请求时后端会重新计算签名比对，不一致就拒绝。要确保安全，得用强算法比如RS256，密钥要保管好，别泄露，服务端还要严格校验签名和过期时间这些字段。

## 4. 如果 JWT 泄露了，别人就能一直用，你怎么处理注销 / 强制下线？

处理JWT泄露后的强制下线或注销，常用的是Redis黑名单机制。用户注销时，把JWT的唯一标识JTI存入Redis黑名单，设置和JWT有效期一致的过期时间。每次请求时，后端先检查JTI是否在黑名单里，在的话就拒绝访问。另外，还可以结合短Token有效期，比如设15分钟，再用刷新Token机制，这样就算Token泄露，有效期也短，风险小。

设计强制下线，一般用Redis黑名单加版本号机制。用户登录时，后端生成JWT，把用户ID和版本号放进payload，同时把版本号存在Redis里。要强制下线，就把Redis里的版本号往上加一。这样用户下次请求，后端一比对，发现JWT里的版本号比Redis里的旧，就会拒绝访问，返回401错误，前端收到后就会让用户退出登录。

## 5. ThreadLocal 存用户信息，你什么时候存、什么时候删？会不会有内存泄漏或线程复用导致用户信息错乱？

在苍穹外卖项目里，用户信息存在ThreadLocal的话，一般是在JWT拦截器的preHandle方法里存，就是解析完token拿到用户ID后，调用BaseContext.setCurrentId存进去。删的话，要在拦截器的afterCompletion方法里，用BaseContext.removeCurrentId删掉，而且最好放在finally块里，确保不管有没有异常都会执行。

要是没删干净，就可能有内存泄漏，因为Tomcat的线程池会复用线程，ThreadLocal里的旧数据没清，新请求用了这个线程就会拿到别人的信息，造成数据错乱。所以一定要记得用完就删，这是最关键的。

## 6. Redis 缓存菜品、套餐

### 6.1 具体缓存了哪些数据？key 怎么设计？

缓存的是菜品和套餐的基本信息，像菜品的ID、名称、价格、图片、分类ID、状态这些，套餐的话还有包含的菜品列表。key的设计是按分类来的，菜品的key是“dish_分类ID”，套餐的是“setmealCache::分类ID”，这样同一个分类下的菜品或套餐就存在一个key里，查的时候方便，更新的时候也能批量删。

## 7. Nginx 负载均衡用的什么策略？为什么选这个？

苍穹外卖项目中Nginx负载均衡默认用的是轮询策略。因为这个策略简单，每个请求按顺序分配到不同服务器，适合服务器性能差不多的情况。轮询能均匀分摊请求，实现负载均衡，而且配置也方便，不需要额外复杂设置。

## 8. 图片上传是前端直传 OSS还是后端转发？为什么这么设计？

苍穹外卖项目里图片上传一般是后端转发的方式。因为项目主要面向学习场景，后端转发实现起来更简单，直接用Spring Boot的MultipartFile接收文件，再调用OSS SDK上传，不需要前端处理复杂的签名逻辑

## 9. 怎么防止别人恶意上传图片

可以从几个方面防止恶意上传，首先后端要校验文件类型，只允许jpg、png等图片格式，拒绝exe、php这类文件。然后限制文件大小，比如设置5MB以内，避免超大文件攻击。还可以用OSS的访问控制，给上传的图片设置只读权限，防止被篡改。另外，上传前通过JWT验证用户登录状态，确保只有合法用户能上传。

## 10. 超时未支付订单，用 Spring Task 扫描，缺点是什么？

Spring Task扫描超时未支付订单的缺点，一是定时任务是固定频率执行的，比如每分钟扫一次，可能导致订单超时后不能立即被处理，有延迟。二是如果订单量很大，每次扫描全表查询未支付订单，会给数据库带来很大压力，影响性能。三是任务不支持分布式，要是项目部署了多台服务器，可能会出现重复扫描、重复处理订单的问题。

## 11. 为什么不用 Redis 过期回调 / RabbitMQ 延迟队列？对比下优缺点。

用Redis过期回调的话，优点是实现简单，直接给订单设置过期时间，到期触发回调处理，但缺点是Redis的过期回调不是100%可靠，可能因为内存回收策略等原因导致回调丢失，而且不适合处理大量订单，容易堆积。RabbitMQ延迟队列的优点是可靠性高，消息不会丢失，支持持久化，适合分布式系统，能精准控制延迟时间，缺点是需要额外部署MQ，增加了系统复杂度，学习和维护成本更高。苍穹外卖作为学习项目，Spring Task足够简单易懂，而Redis过期回调可靠性不足，RabbitMQ又太复杂，所以没选后两者。

## 12. 关单时要回滚库存，你是怎么保证事务 / 一致性的？

关单回滚库存时，通常用本地事务保证一致性。在处理超时订单的方法上加上@Transactional注解，先查询订单包含的菜品/套餐及数量，然后更新库存表，增加对应数量，最后修改订单状态为“已取消”。如果库存更新或订单状态修改失败，事务会自动回滚，确保库存和订单状态要么都改，要么都不改。对于分布式场景，可能需要分布式事务，但苍穹外卖作为单体项目，本地事务足够满足需求。

## 13. 你 WebSocket 是单机还是集群？集群下怎么保证消息能发给对应商家？

苍穹外卖作为单体项目，WebSocket通常是单机部署的。如果是集群部署，多个服务器节点上的WebSocket连接是独立的，消息可能发不到目标商家所在的节点。这时候可以用“消息中间件转发”方案，比如引入RabbitMQ，商家连接WebSocket时，把商家ID和所在服务器节点信息存在Redis里，当需要发消息时，先查Redis找到商家连接的节点，再通过MQ把消息发给对应节点，由该节点的WebSocket推送给商家。

## 14. 高并发下大量连接进来，怎么防止 OOM？

可以通过设置WebSocket连接的最大数量限制，比如用@OnOpen注解里的逻辑判断当前连接数，超过就拒绝新连接。还能调整JVM参数，增大堆内存，但更重要的是优化连接管理，比如给空闲连接设置超时时间，自动断开长时间不通信的连接。另外，用NIO非阻塞IO模型，WebSocket本身基于NIO，能高效处理更多连接，避免每个连接占用独立线程导致资源耗尽。

## 15. 断线重连、消息丢失你怎么处理？

可以在前端断开连接时，将未发送成功的消息暂存在本地，重连成功后重新发送。后端这边可以用消息中间件，比如RabbitMQ，当WebSocket发送消息失败时，把消息存入MQ，重连后从MQ拉取消息再推送。另外，给每个商家维护一个消息队列，断线期间的消息先存在队列里，重连后按顺序补发，确保消息不丢失。

## 16. 催单功能是怎么设计的？会出现重复催单、频繁催单吗？怎么限流？

催单功能一般是用户在订单页面点击“催单”，前端发送请求到后端，后端通过WebSocket给对应商家推送催单消息，同时记录催单时间。为了防止重复或频繁催单，可以用Redis做限流，比如给每个订单ID设置一个key，催单后设置5分钟过期，下次催单时先查Redis，若key存在就拒绝，提示“请勿频繁催单”，过期后才能再次催单。
