## 二、Redis 持久化机制

### RDB（Redis DataBase）快照

RDB 是通过生成某一时刻的数据快照来实现持久化的，可以在特定时间间隔内保存数据的快照。  
适合灾难恢复和备份，能生成紧凑的二进制文件，但可能会在崩溃时丢失最后一次快照之后的数据。

### AOF（Append Only File）日志

AOF 通过将每个写操作追加到日志文件中实现持久化，支持将所有写操作记录下来以便恢复。  
数据恢复更为精确，但文件体积较大，重写时可能会消耗更多资源。

> Redis 4.0 新增了 RDB 和 AOF 的**混合持久化机制**。

## 三、MySQL 索引底层

### 从数据结构角度分类


| 索引类型                     | 说明                                                    |
| ------------------------ | ----------------------------------------------------- |
| **B+ 树索引**               | 树形结构存储数据，支持范围查询（BETWEEN、>、<）和精确查询（=），数据有序存储，便于排序和聚合操作 |
| **哈希索引**                 | 基于哈希表，仅支持等值查询（=），查询速度极快 O(1)，但不支持范围查询和排序              |
| **倒排索引**（全文索引 Full-Text） | 通过分词建立词与文档的映射，支持模糊匹配和自然语言搜索                           |
| **R 树索引**（多维空间树）         | 多维数据结构，支持地理空间数据查询（如距离计算、区域覆盖）                         |


### 从 InnoDB B+ 树索引角度分类

- **聚簇索引（Clustered Index）**：主键索引即聚簇索引，叶子节点存储完整数据行，数据按主键顺序物理存储。
- **非聚簇索引（Non-clustered Index）**：叶子节点仅存储索引列和主键值，查询完整数据需"回表"到聚簇索引。

### 从索引性质角度分类

- **主键索引**：唯一且非空，每个表仅一个，InnoDB 中作为聚簇索引。
- **唯一索引**：确保列值唯一，允许 NULL 值（但只能有一个 NULL）。
- **联合索引**：多列组合索引，遵循最左前缀原则。
- **普通索引**（二级索引、辅助索引）：无唯一性约束，仅加速查询。
- **全文索引**：支持文本内容的模糊搜索和关键词匹配。
- **空间索引**：专为地理数据设计，支持 GIS 查询（如 ST_Distance）。

## 四、聚簇索引与非聚簇索引

**聚簇索引：**

- 索引叶子节点存储的是数据行，可以直接访问完整数据。
- 每个表只能有一个聚簇索引，通常是主键索引，适合范围查询和排序。

**非聚簇索引：**

- 索引叶子节点存储的是数据行的主键和对应的索引列，需通过主键才能访问完整的数据行。
- 一个表可以有多个非聚簇索引（称之为非主键索引、辅助索引、二级索引），适用于快速查找特定列的数据。

## 五、索引失效

### 1. SQL 写法与底层逻辑冲突（破坏 B+ Tree 有序性）

此类问题最为常见，本质是查询条件让底层的 B+ Tree 失去了"二分查找"的快速定位能力。

- **违背最左前缀原则**：跳过联合索引前导列，或遇到范围查询（如 >、<、BETWEEN、`LIKE "abc%"`）导致后续列中断精确定位，降级为范围扫描加过滤。
- **对索引列进行加工**：在 WHERE 左侧对索引列进行数学计算或应用函数，导致原始数据发生逻辑改变，在索引树中呈现无序状态。
- **隐式类型转换（隐蔽且致命）**：当"字符串类型的列"去比较"数字类型的值"时，MySQL 会默认在列上套用转换函数，直接破坏树的有序性。
- **LIKE 模糊查询前置通配符**：如 `LIKE "%abc"`，前缀字符的不确定性使得优化器无法锁定扫描区间的起始点。
- **ORDER BY 排序陷阱**：排序列未命中索引、排序方向与索引结构不一致等触发额外的内存或磁盘排序（Using filesort）。

### 2. 优化器的成本决策（基于 I/O 成本妥协）

此类问题并非索引本身不可用，而是 MySQL 优化器经过计算后，认为"不走普通索引"整体开销反而更小。需要特别说明的是：优化器选择全表扫描或回表查询，往往是正确的成本决策，而非"性能问题"。

- **回表查询是正常现象**：当查询需要非索引覆盖的字段时，回表是不可避免的正常操作。只有当回表次数过多（如命中数据量超过 20%~30%）且存在更优的全表扫描方案时，才需要关注。
- **全表扫描可能是最优选择**：当索引选择率低（命中数据量大）时，顺序 IO 的全表扫描往往比随机 IO 的索引回表更高效。这不是索引"失效"，而是优化器选择了更优的执行路径。
- **SELECT  的场景权衡**：优先 SELECT 需要的字段，能命中覆盖索引最好。如果需要大量非索引字段且回表不可避免，不必教条地"省字段"——当需要大部分字段时，代码可读性可能比"少传几个字段"的微优化更重要。
- **OR 条件导致全表扫描**：只要 OR 连接的任意一侧条件没有对应索引，就会触发全表扫描。即使两侧都有索引，若 Index Merge（索引合并）的预期成本过高，依然会被放弃。
- **IN 列表过长引发估算失真**：当 IN 列表长度超过系统阈值（默认 200）时，优化器会从精准的深入探测（Index Dive）切换为粗略的统计估算，极易因统计信息陈旧而产生执行成本的误判。

## 六、MySQL 执行一条 SQL

```sql
-- 插入单条
INSERT INTO `user` (username, email, age) VALUES ('张三', 'zhangsan@qq.com', 25);

-- 批量插入
INSERT INTO `user` (username, email, age) VALUES
    ('李四', 'lisi@qq.com', 30),
    ('王五', 'wangwu@qq.com', 28);

-- 存在则更新，不存在则插入
INSERT INTO `user` (email, username, age)
VALUES ('zhangsan@qq.com', '张三new', 26)
ON DUPLICATE KEY UPDATE username = '张三new', age = 26;

-- 普通查询
SELECT id, username, email FROM `user` WHERE status = 1;

-- 条件 + 排序 + 分页
SELECT * FROM `user`
WHERE age BETWEEN 20 AND 30
ORDER BY created_at DESC
LIMIT 10 OFFSET 0;

-- 模糊查询（后缀%可走索引）
SELECT * FROM `user` WHERE username LIKE '张%';

-- 更新
UPDATE `user` SET age = 26, status = 1 WHERE id = 1;

-- 批量更新（用 CASE WHEN）
UPDATE `user`
SET age = CASE id
    WHEN 1 THEN 26
    WHEN 2 THEN 31
    WHEN 3 THEN 29
END
WHERE id IN (1, 2, 3);

-- 删除（推荐逻辑删除）
UPDATE `user` SET status = 0 WHERE id = 1;

-- 物理删除
DELETE FROM `user` WHERE id = 1;
```

## 七、TCP 与 UDP 区别


| 场景      | TCP 的做法                                 | UDP 的做法                                 |
| ------- | --------------------------------------- | --------------------------------------- |
| 发消息前    | 先"打招呼"（三次握手），确认对方准备好了才发                 | 直接发，不管对方在不在、有没有准备好                      |
| 发的过程中   | 每发一批数据，等对方说"收到了"才继续（确认机制）；如果超时没收到回复，就重发 | 发完就不管，丢了也不重发                            |
| 数据量大时   | 自动拆分小包，对方收到后再拼起来（拆包/粘包处理）               | 不管大小，直接发（如果超过网络限制，可能被底层拆分，但 UDP 本身不管拼接） |
| 对方忙不过来时 | 主动放慢速度（流量控制、拥塞控制），避免对方"累死"              | 不管对方忙不忙，照发无误，可能导致对方接收的数据"溢出"丢失          |
| 发完后     | 会"道别"（四次挥手），确保双方都处理完                    | 发完就断，不打招呼                               |

## 八、TCP 为什么可靠

> 序号 + 确认应答 + 超时重传 + 流量控制 + 拥塞控制


| 机制   | 解决的问题    | 怎么解决           |
| ---- | -------- | -------------- |
| 序号   | 乱序、重复    | 按序号重排，丢弃重复     |
| 确认应答 | 数据丢失     | 收到就回 ACK，没收到不回 |
| 超时重传 | 丢包没响应    | 超时没收到 ACK 就重发  |
| 流量控制 | 接收方处理不过来 | 接收方告知窗口大小      |
| 拥塞控制 | 网络链路被打爆  | 慢启动 + 动态调整窗口   |

## 九、消息队列作用

### 1. 异步 — 提升响应速度

用户下单后，核心步骤同步执行，非核心步骤丢进队列异步处理：

- 扣库存 + 创建订单（同步，50ms + 50ms）
- 发短信、加积分、推优惠券（异步，丢给队列）

> 响应时间从 500ms → 100ms，用户体验大幅提升

### 2. 解耦 — 降低系统依赖

- **没有 MQ**：订单服务直接调用下游，任意一个挂了，下单失败
- **有了 MQ**：订单服务只发一条消息，下游各自订阅消费，互不影响

> 新增下游服务无需改订单代码，扩展零成本

### 3. 削峰 — 保护系统不崩

双十一零点，每秒 10 万请求：

- 请求先进消息队列缓冲
- 后端按自己处理能力，每秒消费 5000 条
- 数据库始终平稳，不会被瞬间流量压垮

## 十、Kafka 怎么保证消息有序性

Kafka **不保证全局有序，只保证同一个 Partition 内有序**。Partition 是一个只追加写的日志文件，只能往后追加，不能插入中间。

**Kafka 有序性三步走：**

1. 相同业务用相同 key → 保证进同一 Partition
2. 开启生产者幂等性 → 防止重试导致乱序
3. 消费者单线程处理 → 保证消费顺序

> 核心：不追求全局有序，追求同一业务局部有序，既保证了顺序，又保留了 Kafka 的高吞吐能力。

## 十三、JVM 内存分配与回收过程

Java 虚拟机运行时数据区分为方法区、堆、虚拟机栈、本地方法栈、程序计数器。

### 1. 怎么判断对象是"垃圾"？（哪些需要回收）

JVM 使用了**可达性分析算法**（Reachability Analysis）。

- **GC Roots（根节点）**：JVM 会找一些必定不能被回收的活跃对象作为起点，比如虚拟机栈（栈帧中的本地变量表）里引用的对象、方法区中类静态属性引用的对象等。
- **链路追踪**：从这些 GC Roots 开始向下搜索，搜索走过的路径称为引用链。能被这条链串起来的对象就是"存活"的。
- **判定生死**：如果一个对象到 GC Roots 没有任何引用链相连（即不可达），它就会被判定为可回收的"垃圾"。

### 2. 内存是怎么划分的？（在哪回收）

堆内存主要分为两大块：

**新生代（Young Generation）：**

- 分为 1 个 Eden 区和 2 个较小且大小相等的 Survivor 区（通常称为 S0 和 S1）。
- 业务代码中新创建的对象通常都在 Eden 区分配。
- 当 Eden 区满了，会触发 Minor GC。存活的对象会被移动到 Survivor 区；每次在 Survivor 区熬过一次 GC，对象的"年龄"就会增加。

**老年代（Old Generation）：**

- 存放生命周期长、熬过多次 Minor GC 的对象（默认年龄达到 15 岁），或者占用内存特别大的大对象。
- 当老年代空间不足时，会触发 Major GC / Full GC。这种 GC 的执行速度通常比 Minor GC 慢得多，并且更容易导致明显的 STW（Stop-The-World），即暂停所有正在执行业务逻辑的用户线程。对于需要高响应速度的底层系统，减少 Full GC 的频率是性能调优的核心目标。

### 3. 用什么方法打扫卫生？（如何回收）


| 算法                      | 说明                                                                      | 缺点                                    |
| ----------------------- | ----------------------------------------------------------------------- | ------------------------------------- |
| **标记-清除（Mark-Sweep）**   | 先标记出所有需要回收的垃圾，然后直接清理掉                                                   | 清理后会产生大量不连续的内存碎片，导致以后分配大对象时找不到足够的连续空间 |
| **标记-复制（Copying）**      | 将可用内存划分为两块，存活对象整齐地复制到另一块，然后一次性清空原来那块。新生代通常使用此算法变种（Eden + 两个 Survivor 区） | 需要额外的空间进行复制担保                         |
| **标记-整理（Mark-Compact）** | 先标记存活对象，然后让所有存活对象都向内存空间的一端移动，最后直接清理掉边界以外的内存。老年代通常使用此算法                  | 移动对象开销较大                              |

## 十四、Full GC 触发机制

**第一，老年代空间不足。**  
这是最常见的原因。对象不断从新生代晋升到老年代，老年代慢慢被填满，当剩余空间不足以存放新晋升的对象时，触发 Full GC。根本原因通常是：大量长期存活的对象、内存泄漏、或者大对象太多。

**第二，空间分配担保失败。**  
Minor GC 之前，JVM 会做一个预检查：老年代的剩余连续空间，是否大于历次 Minor GC 晋升到老年代的平均大小？如果不满足，JVM 认为这次 Minor GC 风险很高，老年代可能装不下，直接触发 Full GC 先腾空间。

**第三，方法区空间不足。**  
JDK8 之后方法区是 Metaspace，存放类的元数据。如果动态生成了大量类，比如 CGLib 代理没有开缓存、大量 JSP 编译，Metaspace 被撑满，也会触发 Full GC。

**第四，显式调用 `System.gc()`。**  
代码里手动调用，一般会触发 Full GC。生产环境强烈建议加 `-XX:+DisableExplicitGC` 禁掉它，因为有些第三方框架比如 RMI 会悄悄调这个。

**第五，CMS 或 G1 的并发失败退化。**  
如果用 CMS，在并发回收老年代的过程中，新生代 Minor GC 产生的晋升对象把老年代撑满了，CMS 来不及回收，会发生 Concurrent Mode Failure，退化成 Serial Old 做一次完整 Full GC，这是最慢的情况。G1 也类似，如果对象分配速度远超回收速度，堆被填满，G1 会退化为 Serial Full GC。

## 十五、分布式事务解决方案

在微服务架构下，一个业务操作跨多个服务、多个数据库，本地事务无法保证跨服务的数据一致性，这就是分布式事务问题。

### 1. 2PC（两阶段提交）

```
协调者                参与者A      参与者B
  │                    │            │
  │── 第一阶段：准备 ──→│            │
  │── 第一阶段：准备 ──────────────→│
  │                    │            │
  │←── 准备好了 ───────│            │
  │←── 准备好了 ────────────────────│
  │                    │            │
  │── 第二阶段：提交 ──→│            │
  │── 第二阶段：提交 ──────────────→│
```

- **第一阶段**：协调者询问所有参与者能不能提交，参与者锁定资源，回复 Yes/No。
- **第二阶段**：所有人都说 Yes，协调者发提交；有一个说 No，全部回滚。

### 2. TCC（Try-Confirm-Cancel）

业务场景：下单扣库存 + 扣余额

```
Try 阶段（预留资源）：
  库存服务：冻结库存 10 件（不直接扣减，只冻结）
  账户服务：冻结余额 100 元

Confirm 阶段（确认执行）：
  库存服务：正式扣减冻结的 10 件
  账户服务：正式扣减冻结的 100 元

Cancel 阶段（回滚释放）：
  库存服务：释放冻结的 10 件
  账户服务：释放冻结的 100 元
```

三阶段处理：

- **Try**：预留必须的资源，执行业务检查。
- **Confirm**：确认执行，提交事务，释放资源。
- **Cancel**：取消操作，释放预留的资源，回滚事务。

**补偿机制**：TCC 通过补偿操作来保证最终一致性，即使在部分参与者失败的情况下也可以进行处理。

### 3. 3PC

```
协调者                    参与者

第一阶段：CanCommit（询问）
  │── 你能提交吗？ ────────→ │
  │ ←── Yes/No ──────────── │
  （只询问，不锁资源）

第二阶段：PreCommit（预提交）
  │── 准备提交，锁定资源 ──→ │
  │ ←── ACK ──────────────── │
  （锁资源，写 undo/redo log）

第三阶段：DoCommit（正式提交）
  │── 正式提交 ────────────→ │
  │ ←── 完成 ──────────────── │
```

### 4. 本地消息表

将需要执行的操作写到本地消息表中，并由一个独立的进程异步地去处理这些消息，适用于需要最终一致性的场景，例如异步扣库存。

### 5. 可靠消息服务（MQ 事务）

利用消息队列来实现分布式事务，如 RocketMQ 的事务消息，在发送消息和执行本地事务时保证一致性。
