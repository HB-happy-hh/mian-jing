# 腾讯面经 4

## Q14. MySQL 的索引有哪些？

按**数据结构**分类：

| 索引类型 | 说明 | 适用场景 |
|----------|------|----------|
| **B+ 树索引** | MySQL 默认，支持范围查询 | 绝大多数场景 |
| **Hash 索引** | 等值查询极快，不支持范围 | Memory 引擎，等值精确查找 |
| **全文索引** | 对文本分词检索 | 文章内容搜索 |

按**物理存储**分类：

| 索引类型 | 说明 |
|----------|------|
| **聚簇索引**（主键索引） | 叶子节点存储完整行数据，一张表只有一个 |
| **非聚簇索引**（二级索引） | 叶子节点存储主键值，查完需要**回表** |

按**字段特性**分类：

- **主键索引**：唯一 + 非空，自动创建聚簇索引
- **唯一索引**：字段值唯一，允许一个 NULL
- **普通索引**：无限制，最基础的索引
- **联合索引**：多列组合，遵循最左前缀原则
- **前缀索引**：对字符串前 N 个字符建索引，节省空间

## Q15. 联合索引中非叶子节点存储的数据有哪些？

联合索引底层是 **B+ 树**，对于 `INDEX(a, b, c)`：

- **非叶子节点**：存储**所有索引列的键值组合**（如 `(a, b, c)` 的值）+ 指向子节点的**页指针**
- **叶子节点**：存储完整的索引键值 + 对应行的**主键值**（用于回表）

非叶子节点只负责导航，不存储行的其他字段数据。排序规则是先按 `a`，再按 `b`，最后按 `c`，这也是最左前缀原则的底层依据。

## Q16. 最左前缀匹配原则

对于联合索引 `INDEX(a, b, c)`，查询时必须**从最左列开始，且不能跳过中间列**，否则索引失效。

```sql
WHERE a = 1                    -- ✅ 命中 a
WHERE a = 1 AND b = 2          -- ✅ 命中 a, b
WHERE a = 1 AND b = 2 AND c=3  -- ✅ 命中 a, b, c
WHERE a = 1 AND c = 3          -- ⚠️  只命中 a，c 无法用索引（跳过了 b）
WHERE b = 2 AND c = 3          -- ❌ 未命中（跳过了 a）
WHERE b = 2                    -- ❌ 未命中
```

**范围查询会截断后续列：**

```sql
WHERE a = 1 AND b > 2 AND c = 3
-- a 命中，b 命中（范围），c 失效（b 用了范围查询后，c 无法继续走索引）
```

**原因**：B+ 树按 `(a, b, c)` 顺序排序，跳过前面的列，后面的列在树上是无序的，无法利用有序性快速查找。

## Q17. 如何判断有没有命中索引？

使用 `EXPLAIN` 命令：

```sql
EXPLAIN SELECT * FROM users WHERE age = 18 AND name = 'Tom';
```

重点关注字段：

| 字段 | 关注点 |
|------|--------|
| **key** | 实际使用的索引，`NULL` 表示未命中 |
| **type** | 访问类型，`ALL` 是全表扫描，`ref`/`range` 是命中索引 |
| **rows** | 预估扫描行数，越小越好 |
| **Extra** | `Using index` 表示覆盖索引；`Using filesort` 表示需额外排序 |

`type` 性能从好到差：`const > eq_ref > ref > range > index > ALL`

常见索引失效场景：对索引列使用函数、隐式类型转换、`LIKE '%xx'` 开头、`OR` 连接非索引列、不遵循最左前缀原则。

## Q19. Redis 如何实现分布式消息队列？

Redis 提供三种方案，各有适用场景：

**① List（简单队列）**

```bash
LPUSH queue "msg"   # 生产者入队
BRPOP queue 0       # 消费者阻塞出队
```

缺点：消息弹出即消失，无 ACK，不支持多消费者。

**② Pub/Sub（发布订阅）**

```bash
SUBSCRIBE channel     # 订阅
PUBLISH channel "msg" # 发布（广播给所有订阅者）
```

缺点：消息不持久化，订阅者离线期间消息全部丢失。

**③ Stream（生产级，推荐）**

Redis 5.0 引入，支持消费者组、ACK 确认、消息持久化、重复消费。

```bash
XADD stream * key value      # 生产者写入
XGROUP CREATE stream g1 $    # 创建消费者组
XREADGROUP GROUP g1 c1 COUNT 10 BLOCK 0 STREAMS stream >  # 消费
XACK stream g1 <message-id>  # ACK 确认
XPENDING stream g1 - + 10    # 查看未确认（处理失败）的消息
```

| | List | Pub/Sub | Stream |
|--|------|---------|--------|
| 消息持久化 | ✅ | ❌ | ✅ |
| ACK 确认 | ❌ | ❌ | ✅ |
| 消费者组 | ❌ | ❌ | ✅ |
| 广播 | ❌ | ✅ | ✅ |

## Q20. 事务注解 @Transactional 及其底层原理

**底层原理：AOP + 动态代理**

Spring 启动时扫描到 `@Transactional` 注解，为该 Bean 生成代理对象（有接口用 JDK 动态代理，无接口用 CGLIB）。调用方注入的是代理对象，方法调用会被 `TransactionInterceptor` 拦截：

```
调用方 → 代理对象 → TransactionInterceptor.invoke()
                        ↓
              1. 解析注解属性（传播行为、隔离级别等）
              2. getConnection()，关闭 autoCommit
              3. 执行目标方法
              4. 无异常 → commit()
                 有异常 → rollback()
              5. 释放连接
```

**常见失效场景：**

| 原因 | 说明 |
|------|------|
| 非 public 方法 | 代理无法拦截 |
| 同类自调用 `this.method()` | `this` 是原始对象，绕过了代理 |
| 异常被 catch 吃掉 | 拦截器感知不到异常，不触发回滚 |
| 受检异常未配置 rollbackFor | 默认只回滚 `RuntimeException` |
| 方法在新线程中执行 | 事务绑定在 ThreadLocal，新线程无法继承 |

## Q23. JVM 相关的命令

**进程查看：**

```bash
jps -l     # 查看所有 Java 进程及主类名
```

**实时监控（jstat）：**

```bash
jstat -gcutil <pid> 1000   # 每秒输出 GC 使用率（Eden/Old/元空间/GC次数）
jstat -gc <pid> 1000       # 详细 GC 数据
```

**内存快照（jmap）：**

```bash
jmap -heap <pid>                              # 堆内存概要
jmap -histo:live <pid> | head -30             # 存活对象统计
jmap -dump:format=b,file=/tmp/heap.hprof <pid> # 生成堆转储文件
```

**线程快照（jstack）：**

```bash
jstack <pid>          # 打印所有线程栈（排查死锁、CPU高）
jstack -F <pid>       # 强制输出（进程无响应时）
```

> 排查 CPU 100%：`top -Hp <pid>` 找到最高线程 → `printf '%x'` 转16进制 → `jstack` 中 grep 该 nid

**参数查看/修改（jinfo）：**

```bash
jinfo -flag MaxHeapSize <pid>      # 查看单个参数
jinfo -flag +PrintGCDetails <pid>  # 动态开启 GC 日志
```

**综合诊断（jcmd，推荐）：**

```bash
jcmd <pid> VM.flags          # 查看所有 JVM 参数
jcmd <pid> Thread.print      # 打印线程（同 jstack）
jcmd <pid> GC.heap_dump /tmp/heap.hprof  # 堆转储
jcmd <pid> GC.run            # 手动触发 GC
```

**常用排查速查：**

| 问题 | 命令 |
|------|------|
| CPU 100% | `top -Hp` + `jstack` grep nid |
| 内存泄漏 | `jmap -histo:live` + MAT 分析 |
| OOM | 配置 `-XX:+HeapDumpOnOutOfMemoryError` |
| 频繁 Full GC | `jstat -gcutil` 观察 O 区使用率 |
| 死锁 | `jstack` 或 Arthas `thread -b` |

## Q24. 手撕：3. 无重复字符的最长子串

详见：[题目描述与图解](../算法/无重复字符的最长子串/题目描述.md)

**核心思路：滑动窗口 + HashMap**

```java
public int lengthOfLongestSubstring(String s) {
    Map<Character, Integer> map = new HashMap<>();
    int maxLen = 0, left = 0;

    for (int right = 0; right < s.length(); right++) {
        char c = s.charAt(right);
        if (map.containsKey(c) && map.get(c) >= left) {
            left = map.get(c) + 1;  // left 只能前进，不能后退
        }
        map.put(c, right);
        maxLen = Math.max(maxLen, right - left + 1);
    }
    return maxLen;
}
```

时间复杂度 O(n)，空间复杂度 O(min(m, n))。
