后端分层架构

Access · Application · Cache · Database

Layer 1 · 接入层
接入层 Access Layer
所有外部流量的第一道入口，负责路由分发、限流、鉴权、SSL 终止。 用户感知到的就是这一层。
DNS / CDN
就近分发静态资源，减少源站压力
Nginx / 网关
反向代理、负载均衡、限流熔断
API Gateway
鉴权、路由、灰度发布、日志采集
路由转发
Layer 2 · 应用层
应用层 Application Layer
核心业务逻辑处理层，无状态水平扩展。 本地缓存（Caffeine）部署在此层，拦截热点读请求。
业务服务
Spring Boot 微服务，处理订单、用户、商品等逻辑
本地缓存
Caffeine，拦截约 70% 热点读，纳秒级响应
消息队列
RocketMQ 异步削峰，解耦下游依赖
水平扩展 无状态 熔断降级 布隆过滤器
未命中本地缓存
Layer 3 · 缓存层
缓存层 Cache Layer
分布式缓存，拦截剩余 ~20% 请求，毫秒级响应。 通过 Lua 脚本保证原子性操作（如库存扣减），防止超卖。
Redis 主从
主写从读，读写分离，支持高并发读
Redis Cluster
数据分片，横向扩展容量与吞吐
Lua 脚本
原子性扣减库存，避免并发超卖
防缓存穿透 防缓存击穿 防缓存雪崩
未命中缓存（仅 ~10%）
Layer 4 · 数据库层
数据库层 Database Layer
最终持久化层，只承接少量穿透请求（约 10%）。 通过读写分离、分库分表应对高并发写入。
MySQL 主库
处理写操作，binlog 同步至从库
MySQL 从库
承接读请求，读写分离降低主库压力
分库分表
ShardingSphere，水平拆分超大表
索引优化 慢查询监控 连接池

请求流向

用户请求 → CDN / Nginx → API Gateway 鉴权 → 业务服务 → 本地缓存 Caffeine → Redis → MySQL

各层拦截率（以 1000 QPS 为例）

CDN
拦截 ~40%
本地缓存
拦截 ~70%
Redis
拦截 ~20%
MySQL
到达 ~10%
