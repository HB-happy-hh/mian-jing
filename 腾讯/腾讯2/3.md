 1.如果我们要去设计一套高并发的票务系统，如何去确保这个系统的高吞吐，并且确保它的稳定性和票不能够超卖？对于这么一个产品，你要怎么去设计？

 我的思路是：层层拦截流量，把真正需要处理的请求降到最低；用 Redis + Lua 做原子库存扣减防超卖；用 MQ 异步削峰保护数据库。
 开票前：
  ① CDN 缓存静态页面
  ② 库存预热到 Redis
  ③ 等候室排队，过滤流量

开票瞬间：
  ④ 网关限流（令牌桶），超出直接拒绝
  ⑤ 用户去重，1秒内只处理一次

抢票核心：
  ⑥ Lua 脚本原子扣减 Redis 库存（第一道防线）
  ⑦ 发 MQ 消息，立刻返回"抢票成功"

异步处理：
  ⑧ 消费 MQ，幂等创建订单
  ⑨ 数据库乐观锁 + 唯一索引（第二三道防线）

后续：
  ⑩ 用户支付，更新订单状态
  ⑪ 15分钟未支付，自动取消，归还库存

5. 你提到在前端调度上要做CDN，这是为了做什么？

用户打开票务网站，浏览器要加载：
  活动页面的 HTML
  CSS 样式文件
  JavaScript 文件
  活动图片（可能很大）
  歌手照片、演出海报...

没有 CDN：
  全国所有用户的这些请求都打到你的服务器
  100万人同时打开页面
  100万个"给我图片"的请求打到北京的服务器
  服务器带宽直接被图片请求占满
  真正的抢票请求反而进不来！
  
  CDN 的做法：
  在全国各地部署很多节点（边缘服务器）
  把静态资源（图片、JS、CSS）缓存到这些节点

CDN = 在用户附近缓存静态资源
      把"给我图片/JS/CSS"这类请求
      拦截在离用户最近的节点解决
      不让这些请求打到源服务器
