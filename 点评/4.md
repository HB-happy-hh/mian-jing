## 1. 为什么要加 Redis 分布式锁？加锁目的是什么？

根本原因：**单机锁（synchronized）在集群环境下失效**。

项目部署了多个服务实例，`synchronized` 只能锁住同一个 JVM 内的线程，跨进程完全无效：

```
用户X的两个请求同时打进来
   ↓               ↓
实例A             实例B
synchronized      synchronized
（A自己的锁）    （B自己的锁）
   ↓               ↓
两把锁互不相干，各自通过 ✗ → 重复下单！
```

加 Redis 分布式锁后，所有实例共用同一个 Redis，锁是全局唯一的：

```
实例A：SETNX lock:userId → 成功，继续执行
实例B：SETNX lock:userId → 失败，返回"不能重复下单"
```

锁的 key 是 `lock:order:{userId}`，**按用户粒度加锁**，不同用户并行，同一用户互斥，性能影响最小。

> 一句话：加 Redis 分布式锁，是为了在集群环境下保证"一人一单"的互斥性，弥补 synchronized 只能锁单机 JVM 的局限。

## 2. 如果锁宕机了（服务挂了），要怎么处理？会引发什么风险？

### 情况一：持锁的服务实例宕机（JVM 崩了）

`finally` 里的 `unlock()` 没执行，锁不会被主动释放。

**Redisson 的应对**：锁设置了 TTL，到期自动释放。看门狗线程随 JVM 一起死，锁最多等 30 秒后自动过期，期间业务短暂不可用，但不会出现数据错误。

### 情况二：Redis 本身宕机（更严重）

**主从架构下的脑裂风险：**

```
① 线程A 在 Master 上抢到锁
② Master 还没把锁同步给 Slave
③ Master 宕机，Slave 被提升为新 Master
④ 新 Master 上没有这把锁！
⑤ 线程B 也抢到"锁" → 两把锁同时存在 → 互斥性被破坏
```

**解决方案：Redlock 红锁算法**

部署 5 个独立 Redis 实例，加锁时必须在多数节点（≥3个）上成功才算持有锁。一个节点宕机，线程B 最多拿到 2 个，< 3，加锁失败，互斥性保证。

| 方案 | 解决的问题 | 代价 |
|------|-----------|------|
| TTL 自动过期 | 服务实例宕机后锁自动释放 | 短暂不可用（30s内） |
| Redlock | Redis 节点宕机后互斥性不丢失 | 需要5个独立 Redis，成本高 |

> 实际项目中：单节点 Redis + Redisson，数据库唯一索引兜底，即使锁偶发失效也不会真正产生重复订单。

## 3. 如何去更新库存？多人购买的冲突问题怎么解决？

### 问题根源：超卖

```
库存剩 1 件
线程A：查库存 → 1，准备扣
线程B：查库存 → 1，准备扣
线程A：扣库存 → 0
线程B：扣库存 → -1 ← 超卖！
```

"查库存"和"扣库存"是两步，不是原子操作。

### 解决：Redis + Lua 脚本

把"判断库存 + 判断一人一单 + 扣库存"合并为一个原子命令：

```lua
local voucherId = ARGV[1]
local userId    = ARGV[2]

local stockKey = 'seckill:stock:' .. voucherId
local orderKey = 'seckill:order:' .. voucherId

-- 判断库存是否充足
if (tonumber(redis.call('get', stockKey)) <= 0) then
    return 1  -- 库存不足
end

-- 判断是否重复下单
if (redis.call('sismember', orderKey, userId) == 1) then
    return 2  -- 重复下单
end

-- 扣减库存 + 记录用户
redis.call('incrby', stockKey, -1)
redis.call('sadd', orderKey, userId)
return 0  -- 成功
```

**为什么 Lua 能保证原子性？**
Redis 是单线程执行命令的，Lua 脚本作为一个不可分割的命令执行，执行过程中不会有其他命令插入。

MySQL 唯一索引 `(user_id, voucher_id)` 作为最后一道兜底保障。

## 4. Redis 集群下 Lua 多 key 的 CROSSSLOT 问题

### 为什么会报错

Redis Cluster 把数据分散到 16384 个 slot，每个 key 按 `CRC16(key) % 16384` 计算所在 slot，不同 key 可能落在不同节点。

Lua 脚本要求所有 key 必须在同一节点，否则报错：
```
CROSSSLOT Keys in request don't hash to the same slot
```

原因：Lua 原子性只对单节点有效，Redis 无法跨节点保证原子性，索性直接禁止。

### 解决：Hash Tag `{}`

**花括号内的部分相同，Redis 只取花括号内的内容计算 slot：**

```
修改前：
  seckill:stock:1001  → CRC16全串 → slot 3200 → 节点1
  seckill:order:1001  → CRC16全串 → slot 9876 → 节点2 ✗

修改后：
  seckill:{1001}:stock → CRC16("1001") → slot 7003 → 节点2
  seckill:{1001}:order → CRC16("1001") → slot 7003 → 节点2 ✓
```

代码只需修改 key 命名：
```java
String stockKey = "seckill:{" + voucherId + "}:stock";
String orderKey = "seckill:{" + voucherId + "}:order";
```

**注意副作用：** 不要用固定字符串（如 `{seckill}`）作为所有 key 的 Hash Tag，否则全部堆到一个节点，造成数据倾斜。应用业务 ID（voucherId）作为 Tag，不同商品天然分散。

## 5. MQ 漏发场景下的状态一致性——项目中的例子

### 漏发发生在哪里

```
① Lua 脚本执行成功（Redis 扣库存、记录用户）
② 发送消息到 MQ           ← 漏发发生在这里
③ 消费者写入 MySQL 订单表
```

### 具体场景

```
Lua 执行成功
  ↓
Redis：stock=9，userId_A 已记录为"已购买"
  ↓
准备发 MQ 消息... 服务器宕机 / 网络抖动
  ↓
消息没发出去！MySQL 无订单记录
```

**状态不一致后果：**
- 库存被多扣1个，其他用户永远少抢一个
- 用户A 再购买，Lua 判断"已购买"直接拒绝，永远买不了

**初版阻塞队列（ArrayBlockingQueue）问题更严重：**
- JVM 宕机，队列在内存里，所有消息全部丢失
- 队列满了，消息直接丢弃

这也是后来换 RabbitMQ 的根本原因。

### 解决方案

**本地消息表（推荐）：**
```
① Lua 成功
② MySQL 写一条 status=PENDING 记录
③ 发 MQ 消息
④ Broker 确认收到 → 更新 status=SENT
⑤ 定时任务扫描 PENDING 超时记录，重新投递
```

## 6. RabbitMQ 如何解决漏发问题（三个环节全覆盖）

### ① 生产者 → Broker：Publisher Confirm

```yaml
spring:
  rabbitmq:
    publisher-confirm-type: correlated
    publisher-returns: true
```

```java
template.setConfirmCallback((correlationData, ack, cause) -> {
    if (!ack) {
        // Broker 没收到：记录日志，重发
        retrySend(correlationData.getId());
    }
});
```

结合本地消息表：Lua 成功后写 PENDING 记录，confirmAck 后更新为 SENT，定时任务补投超时的 PENDING 记录。

### ② Broker 存储：消息持久化

队列和消息都要设置持久化，防止 Broker 宕机丢数据：
```java
// 队列持久化
new Queue("seckill.order.queue", true);

// 消息持久化
msg.getMessageProperties().setDeliveryMode(MessageDeliveryMode.PERSISTENT);
```

### ③ 消费者：手动 ACK

```yaml
spring.rabbitmq.listener.simple.acknowledge-mode: manual
```

```java
@RabbitListener(queues = "seckill.order.queue")
public void handleOrder(VoucherOrder order, Channel channel,
                        @Header(AmqpHeaders.DELIVERY_TAG) long tag) {
    try {
        orderService.createOrder(order);
        channel.basicAck(tag, false);       // 成功确认
    } catch (Exception e) {
        channel.basicNack(tag, false, true); // 失败重入队
    }
}
```

重发必然带来重复消费，用数据库唯一索引 `(user_id, voucher_id)` 做幂等兜底，捕获 `DuplicateKeyException` 即可。

| 环节 | 机制 | 作用 |
|------|------|------|
| 生产者 → Broker | Publisher Confirm + 本地消息表 | 漏发感知并重投 |
| Broker 存储 | 队列+消息双持久化 | 宕机不丢消息 |
| Broker → 消费者 | 手动 ACK + basicNack 重入队 | 消费失败自动重试 |

## 7. 父子进程与 Copy-On-Write（COW）

### 没有 COW 的问题

fork 时直接复制父进程全部内存，内存瞬间翻倍，耗时极长，大量复制是浪费。

### COW 核心思想

> 不复制，共享。谁要改，谁再复制那一页。

### fork 过程

**fork 刚发生：** 父子进程页表都指向同一块物理内存，全部标记为只读。内存用量不变。

**子进程尝试写某页时：**
1. CPU 触发写保护异常（Page Fault）
2. OS 复制该页为新页
3. 子进程页表指向新页，标记可写
4. 原页归父进程，标记恢复可写

只复制被修改的那一页，其余继续共享。

### COW 与 Redis 的关系

Redis 做 RDB 持久化时 fork 子进程：

```
主进程（父）          子进程
  │ fork()              │
  │ 继续处理写请求       │ 遍历内存，写 RDB 文件
  │                     │
  │ 客户端写了 key A     │
  ↓                     ↓
触发 COW：           子进程看到 fork 时刻的
复制 key A 所在页    旧数据（快照），不受影响
```

这就是 RDB 能生成"某一时刻快照"的原因。主进程写入量越大，COW 触发越多，内存增长越明显，最坏情况接近翻倍——所以 Redis 建议预留当前内存 **1 倍冗余**，防止 OOM。

## 8. TCP 流量控制 vs 拥塞控制

```
流量控制：发送方 和 接收方 之间的协商 → 解决"接收方处理不过来"
拥塞控制：发送方 和 网络 之间的协商  → 解决"网络中间路由器撑不住"

实际发送窗口 = min(rwnd, cwnd)
               ↑流量控制  ↑拥塞控制
```

### 流量控制：滑动窗口

接收方在每个 ACK 里携带 `rwnd`（接收窗口大小），告知发送方缓冲区剩余空间，发送方不能超发。

`rwnd=0` 时发送方停止，靠**坚持定时器**定时发 1 字节探测，防止 ACK 丢失导致死锁。

### 拥塞控制：四个算法

**① 慢启动：** cwnd 从 1 开始指数增长（每收到一个 ACK，cwnd+1），直到达到 ssthresh。

**② 拥塞避免：** cwnd 超过 ssthresh 后改为线性增长（每轮+1 MSS），慢慢探测网络极限。

**③ 快速重传：** 收到 3 个重复 ACK，立刻重传丢失的包，不等超时定时器。

**④ 快速恢复：**
- 收到 3 个重复 ACK（网络还好）：`ssthresh = cwnd/2`，`cwnd = ssthresh`，进入拥塞避免
- 超时丢包（网络很差）：`ssthresh = cwnd/2`，`cwnd = 1`，从慢启动重来

```
cwnd
 ^
         ssthresh
──────────┐ 指数→ 线性
          │           丢包(3ACK)
          *         *←────── cwnd=ssthresh，不归1
         *         *
        *          线性增长继续
慢启动  拥塞避免    快速恢复+拥塞避免
```

| | 流量控制 | 拥塞控制 |
|--|---------|---------|
| **目的** | 保护接收方 | 保护网络 |
| **控制量** | rwnd | cwnd |
| **信息来源** | 接收方 ACK | 丢包/重复ACK（推断）|

## 9. LeetCode 93：复原 IP 地址

**题目：** 给定字符串 `"25525511135"`，输出所有合法 IP 地址。

**答案：**
```
255.255.11.135
255.255.111.35
```

### 思路：回溯（DFS）

把字符串切成 4 段，每段满足：长度 1~3、值 0~255、无前导零。

```java
class Solution {
    List<String> result = new ArrayList<>();

    public List<String> restoreIpAddresses(String s) {
        dfs(s, 0, new ArrayList<>());
        return result;
    }

    private void dfs(String s, int index, List<String> segments) {
        // 切了4段且用完所有字符 → 合法IP
        if (segments.size() == 4 && index == s.length()) {
            result.add(String.join(".", segments));
            return;
        }
        if (segments.size() == 4 || index == s.length()) return;

        for (int len = 1; len <= 3; len++) {
            if (index + len > s.length()) break;

            String seg = s.substring(index, index + len);

            // 剪枝：前导零
            if (seg.length() > 1 && seg.charAt(0) == '0') break;
            // 剪枝：超过255
            if (Integer.parseInt(seg) > 255) break;

            segments.add(seg);
            dfs(s, index + len, segments);
            segments.remove(segments.size() - 1); // 回溯
        }
    }
}
```

### 剪枝条件

| 剪枝条件 | 说明 |
|---------|------|
| 4段了但字符没用完 | 非法，直接返回 |
| 字符用完了但不够4段 | 非法，直接返回 |
| 前导零（"01"、"001"） | break，更长的也不行 |
| 值 > 255 | break，更长的更不行 |

**时间复杂度：** O(3⁴) = O(81)，近似 O(1)
