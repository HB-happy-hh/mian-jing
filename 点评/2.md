# 黑马点评面试题解答

## 1. 项目背景与业务介绍

黑马点评是一个类似大众点评的本地生活服务平台，核心业务包括：

- **用户模块**：注册登录（短信验证码 + Session/Token）、用户信息管理
- **商户模块**：商户查询、按类型/地理位置查询附近商户（GEO）
- **优惠券秒杀模块**：库存管理、一人一单、防超卖、异步下单
- **社交模块**：博客发布、点赞（Set）、关注/共同关注（Set）、好友动态 Feed 流（推模式）
- **缓存优化**：多级缓存（本地 Caffeine + Redis）、缓存穿透/击穿/雪崩处理

---

## 2. 超卖问题怎么解决

超卖的根本原因是"查库存"和"扣库存"两步操作不是原子的，多线程并发时会出现库存已经为 0 但还继续扣的情况。

**解决方案：Lua 脚本 + Redis 原子操作**

```lua
-- 秒杀 Lua 脚本核心逻辑
local voucherId = ARGV[1]
local userId = ARGV[2]

local stockKey = 'seckill:stock:' .. voucherId
local orderKey = 'seckill:order:' .. voucherId

-- 1. 判断库存是否充足
if (tonumber(redis.call('get', stockKey)) <= 0) then
    return 1  -- 库存不足
end

-- 2. 判断是否已经下单（一人一单）
if (redis.call('sismember', orderKey, userId) == 1) then
    return 2  -- 重复下单
end

-- 3. 扣减库存、记录用户
redis.call('incrby', stockKey, -1)
redis.call('sadd', orderKey, userId)
return 0  -- 秒杀成功
```

Lua 脚本在 Redis 单实例下具有原子性，整个判断 + 扣减过程不会被打断，彻底解决超卖。

## 3. 秒杀优化中异步下单的内容

**原同步流程**：用户请求 → 查 Redis 库存 → 扣库存 → 写数据库订单 → 返回

写数据库是慢操作，高并发下响应慢。

**异步优化流程**：

1. 用户请求到达，执行 Lua 脚本在 Redis 中完成资格校验（库存、一人一单）
2. 校验通过后，将订单信息封装成消息放入**阻塞队列**（或 MQ）
3. **立即返回订单 ID** 给用户，告知下单成功
4. 后台独立线程（消费者）从队列取出消息，**异步写入数据库**

```
用户 → Lua校验(Redis) → 入队 → 立即返回
                              ↓
                    后台线程异步写DB
```

这样将耗时的 DB 操作从主链路中剥离，响应时间大幅降低。

---

## 4. 一人一单在哪里判断，记录在哪里

- **判断在哪里**：在 Redis 中通过 Lua 脚本判断（原子性保证），而不是查数据库
- **记录在哪里**：Redis 中用 `Set` 结构记录，Key 为 `seckill:order:{voucherId}`，Value 是下过单的 `userId` 集合。用 `SISMEMBER` 检查、`SADD` 记录，O(1) 复杂度

数据库中订单表也有 `(user_id, voucher_id)` 的唯一索引作为兜底保障，但主要判断逻辑在 Redis。

---

## 5. 为什么要用异步下单


| 同步下单            | 异步下单                   |
| --------------- | ---------------------- |
| 主线程等待 DB 写入     | 主线程只操作 Redis，极快        |
| DB 是性能瓶颈        | DB 操作异步化，不阻塞用户         |
| 高并发直接打到 DB      | 队列起到削峰作用               |
| TPS 受 DB 写入速度限制 | TPS 取决于 Redis 速度，提升数量级 |


核心原因：**Redis 操作比 DB 快 10~100 倍**，将慢操作异步化可以大幅提升吞吐量，同时通过队列实现削峰。

## 6. 阻塞队列中的订单下单失败了怎么办

这是用 JDK `ArrayBlockingQueue` 实现时的痛点：

**问题**：

- 阻塞队列基于 JVM 内存，服务宕机后队列数据**全部丢失**
- 消费者处理失败时，没有重试机制
- 队列满了会阻塞或丢消息

**解决方案**（引出 MQ 的必要性）：

1. **引入 RabbitMQ/RocketMQ**：消息持久化到磁盘，宕机不丢失
2. **消费者 ACK 机制**：处理成功才确认消费，失败自动重试
3. **死信队列**：多次重试仍失败的消息转入死信队列，人工介入处理
4. **幂等设计**：数据库唯一索引兜底，重复消费不会重复下单

## 7. 热 Key 问题怎么解决（Redis 扛不住怎么办）

热 Key 指某个 Key 被极高频访问，导致单个 Redis 节点压力过大。

**解决方案（分层处理）**：

**① 本地缓存（一级缓存）**

- 用 Caffeine 在 JVM 内存中缓存热点数据
- 请求优先查本地缓存，命中则完全不访问 Redis
- 适合读多写少、允许短暂不一致的数据

**② Key 分散（Redis 层）**

- 将热 Key 复制为多份：`hot:key:0`、`hot:key:1`...`hot:key:N`
- 请求时随机选择一个副本，分散到多个 Redis 节点

**③ 读写分离 + 集群扩容**

- Redis 主从 + 集群模式，读请求打到多个从节点

**④ 限流 + 降级**

- 对热 Key 请求做限流（Sentinel/Guava RateLimiter）
- 降级时返回兜底数据

项目中的实现是 **Caffeine 本地缓存 + Redis 二级缓存**，大部分热点请求在本地缓存命中，不会到达 Redis。

---

## 8. 多级缓存是怎么做的

```
请求
 ↓
本地缓存（Caffeine）── 命中 → 返回
 ↓ 未命中
Redis 缓存 ── 命中 → 写回本地缓存 → 返回
 ↓ 未命中
数据库 → 写入 Redis + 本地缓存 → 返回
```

**关键点**：

- **Caffeine**：设置最大容量 + 过期时间（TTL 较短，如 5 分钟）
- **Redis**：TTL 较长（如 30 分钟），用于跨节点共享
- **一致性**：数据更新时先更新 DB，再删除 Redis，本地缓存靠 TTL 自然过期（或用 Canal 监听 binlog 主动失效）
- **集群环境下本地缓存一致性**：通过 Redis Pub/Sub 广播失效通知，所有节点收到后清除本地缓存

## 10. Java 中保证线程安全的方法


| 方式                        | 说明                  |
| ------------------------- | ------------------- |
| `synchronized`            | 内置锁，修饰方法/代码块        |
| `ReentrantLock`           | 显式锁，支持超时、公平锁、可中断    |
| `volatile`                | 保证可见性和禁止指令重排，不保证原子性 |
| `ThreadLocal`             | 变量线程隔离，各线程独立副本      |
| 原子类（`AtomicInteger` 等）    | CAS 无锁操作            |
| 不可变对象（`final`、`String`）   | 天然线程安全              |
| 并发容器（`ConcurrentHashMap`） | 线程安全的集合实现           |

## 11. volatile 能保证线程安全吗

**不能完全保证**。volatile 只提供两个保证：

1. **可见性**：一个线程修改变量，其他线程立即可见（强制从主内存读）
2. **有序性**：禁止指令重排序（通过内存屏障）

**不保证原子性**：经典反例是 `i++`，它是"读-改-写"三步操作，volatile 无法保证这三步的原子性，多线程下仍会出现竞争问题。

**适用场景**：状态标志位（boolean），或配合 `double-checked locking` 实现单例。

## 12. JVM 垃圾回收机制


| GC 类型                   | 特点                   | 优点             | 缺点                |
| ----------------------- | -------------------- | -------------- | ----------------- |
| **Serial GC**           | 单线程，STW              | 简单，低内存占用       | 停顿时间长，不适合并发场景     |
| **Parallel GC**（JDK8默认） | 多线程并行，追求吞吐量          | 高吞吐            | STW 停顿时间不可控       |
| **CMS**                 | 并发标记，减少 STW          | 低停顿            | 内存碎片，浮动垃圾，CPU 占用高 |
| **G1**（JDK9+默认）         | Region 分区，优先回收垃圾最多区域 | 停顿可预测，适合大堆     | 内存占用较大            |
| **ZGC/Shenandoah**      | 几乎全程并发               | 停顿 < 10ms，超低延迟 | 吞吐量略低，内存消耗大       |

## 13. HashMap 详解

**数据结构**：JDK8 中是 **数组 + 链表 + 红黑树**

- 默认初始容量 16，负载因子 0.75

**put 过程（寻址）**：

```
1. 计算 key 的 hashCode，再做扰动：hash = h ^ (h >>> 16)
2. 计算下标：index = hash & (capacity - 1)（等价于取模但更快）
3. 若该位置为空，直接插入
4. 若不为空（hash 冲突），链表追加；若链表长度 >= 8 且数组长度 >= 64，转红黑树
5. 若 key 已存在，更新 value
6. 若 size > capacity * loadFactor，触发扩容
```

**扩容**：

- 容量翻倍（×2），重新计算每个元素的位置
- JDK8 优化：元素新位置 = 原位置 或 原位置 + 旧容量（通过高位 bit 判断，无需重新 hash）

**Hash 冲突**：HashMap 用**链地址法**（拉链法），冲突元素挂在同一个桶的链表/红黑树上

## 14. 其他处理 Hash 冲突的方式


| 方法            | 说明                     |
| ------------- | ---------------------- |
| **链地址法（拉链法）** | HashMap 用的，冲突元素用链表/树存储 |
| **开放地址法**     | 冲突时在数组中寻找下一个空位         |
| **再哈希法**      | 冲突时用第二个哈希函数重新计算位置      |
| **建立公共溢出区**   | 冲突元素统一放到单独的溢出区域        |

## 15. 线性探测法的几种方式及优缺点

线性探测法属于**开放地址法**的一种，有以下几种变体：


| 方式                          | 探测步骤                      | 优点        | 缺点                 |
| --------------------------- | ------------------------- | --------- | ------------------ |
| **线性探测（Linear Probing）**    | `(hash + i) % n`，逐个往后找    | 简单，缓存友好   | "聚集"问题严重，性能下降快     |
| **二次探测（Quadratic Probing）** | `(hash + i²) % n`，跳跃式查找   | 减少聚集      | 不能保证遍历所有位置，可能找不到空槽 |
| **双重哈希（Double Hashing）**    | `(hash1 + i * hash2) % n` | 分布均匀，聚集最少 | 计算复杂，两个哈希函数设计有要求   |

## 手撕：236. 二叉树的最近公共祖先

```java
public TreeNode lowestCommonAncestor(TreeNode root, TreeNode p, TreeNode q) {
    // base case: 遇到 null 或找到 p/q 之一就返回
    if (root == null || root == p || root == q) {
        return root;
    }

    TreeNode left  = lowestCommonAncestor(root.left,  p, q);
    TreeNode right = lowestCommonAncestor(root.right, p, q);

    // 左右都找到，说明 root 就是 LCA
    if (left != null && right != null) return root;

    // 只有一侧找到，返回那一侧
    return left != null ? left : right;
}
```

**思路**：后序遍历，递归收集左右子树的结果：

- 左右都非 null → 当前节点就是 LCA
- 只有一侧非 null → LCA 在那一侧（或那侧本身就是 LCA）
- 都为 null → 该子树中不包含 p 或 q

时间复杂度 O(n)，空间复杂度 O(h)（h 为树高）。
