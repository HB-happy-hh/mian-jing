## 17. 为什么用两级缓存，只用 Redis 行不行

**只用 Redis 的问题**：


| 问题         | 说明                                   |
| ---------- | ------------------------------------ |
| 网络开销       | 每次请求都要经过网络 IO 访问 Redis，延迟在 0.5ms~2ms |
| 带宽瓶颈       | 热点数据被高频访问，Redis 服务器网卡带宽可能成为瓶颈        |
| Redis 单点压力 | 即使是集群，热 Key 始终落在同一个节点上，无法分散          |
| 连接数限制      | Redis 连接数有上限，高并发下连接池可能耗尽             |


**加入 Caffeine 本地缓存后**：

```
热点数据直接从 JVM 内存读取，耗时 < 1μs（微秒）
完全不走网络，不占用 Redis 连接，不消耗带宽
```

**为什么有这个考虑**：
秒杀场景下，商品信息、库存预热数据是典型的**读多写少**数据，短时间内数据不变，但访问量极大。本地缓存天然适合这种场景，能把 Redis 的压力降低 80% 以上。代价是牺牲一点一致性（本地缓存有短暂的 TTL 过期延迟），这在秒杀场景下是完全可以接受的。

## 18. 削峰解耦后吞吐量提升了多少

**标准回答思路**（没有真实压测数据时的诚实答法）：

没有在生产环境做过严格压测，但可以从理论和对比角度说明：

- **同步写 DB 的瓶颈**：MySQL 单机 TPS 约 1000~3000（写操作），受磁盘 IO 限制
- **异步化后的瓶颈**：变为 Redis 的处理能力，单节点 QPS 可达 10 万+
- **理论提升**：主链路响应时间从 50ms+ 降低到 5ms 以内，吞吐量提升约 **10~30 倍**

如果面试官追问压测数据，直接说：**"这个是课程项目，没有做过生产级压测，但我了解 JMeter/wrk 的压测方法，如果上生产我会通过压测工具采集具体数据。"** 诚实比编数字更好。

## 19. Redis 集群的几种方式 + 主观下线/客观下线 + 哨兵选主

### 集群的三种方式


| 方式                 | 说明                           | 适用场景       |
| ------------------ | ---------------------------- | ---------- |
| **主从复制**           | 一主多从，读写分离，主节点写，从节点读          | 读多写少，基础高可用 |
| **哨兵模式（Sentinel）** | 在主从基础上加哨兵节点，自动监控+故障转移        | 自动化高可用     |
| **Cluster 集群**     | 数据分片到 16384 个 slot，多主多从，横向扩容 | 数据量大，高并发   |


### 主观下线（SDOWN）vs 客观下线（ODOWN）

- **主观下线**：某一个哨兵 ping 主节点超时，**单独认为**主节点挂了，记为主观下线
- **客观下线**：多个哨兵都认为主节点挂了，达到 `quorum`（法定人数）阈值，**集体确认**下线，记为客观下线

> 设计目的：防止网络抖动导致误判，单个哨兵的判断不可信，需要多数哨兵达成共识。

### 领导哨兵怎么选（Raft 算法）

1. 发现主节点客观下线的哨兵，**向其他哨兵拉票**，请求成为 Leader
2. 每个哨兵在本轮只能投一票，**先到先得**
3. 获得超过半数票的哨兵当选 Leader
4. Leader 负责执行故障转移

### 领导哨兵按什么规则选新主节点

按以下优先级依次筛选：

```
1. 过滤掉不健康的从节点（断线、响应慢）
2. 优先选 slave-priority（优先级配置）最小的
3. 优先选复制偏移量（offset）最大的（数据最新的）
4. 以上都相同，选 runId 字典序最小的
```

## 20. 为什么不用 Redis 原生 setnx 锁，而用 Redisson 分布式锁

### setnx 锁的实现

```java
// 加锁
String value = UUID.randomUUID().toString();
Boolean success = redisTemplate.opsForValue()
    .setIfAbsent("lock:key", value, 10, TimeUnit.SECONDS);

// 解锁（必须判断是自己的锁再删）
if (value.equals(redisTemplate.opsForValue().get("lock:key"))) {
    redisTemplate.delete("lock:key");  // 判断和删除不是原子的！
}
```

### setnx 的四大缺陷

**① 不可重入**
同一个线程如果两次尝试获取同一把锁，第二次会被自己阻塞，造成死锁。

**② 锁超时释放，但业务还没执行完**

```
业务执行需要 15 秒，但锁过期时间设了 10 秒
→ 10 秒后锁自动释放
→ 其他线程拿到锁进入临界区
→ 两个线程同时执行，线程安全被破坏
```

设置过长怕死锁，设置过短怕业务没跑完，这个时间很难估。

**③ 解锁非原子（判断 + 删除两步）**

```
线程A：get("lock") == 自己的值  → 准备删除
    ← 此时锁恰好过期，线程B抢到锁 →
线程A：delete("lock")  → 把线程B的锁删了！
```

需要用 Lua 脚本保证判断+删除原子性，但自己实现容易出错。

**④ 不支持阻塞等待**
`setnx` 失败只能立即返回或手写自旋重试，没有等待队列机制。

### Redisson 分布式锁的优点


| 特性        | setnx 手写 | Redisson                 |
| --------- | -------- | ------------------------ |
| **可重入**   | 不支持      | 支持（Hash 结构记录重入次数）        |
| **自动续期**  | 不支持      | 看门狗机制，业务未完自动续期           |
| **解锁原子性** | 需自己写 Lua | 内置 Lua 脚本，开箱即用           |
| **阻塞等待**  | 需自己自旋    | `tryLock(waitTime)` 支持等待 |
| **公平锁**   | 不支持      | `getFairLock()` 支持       |
| **读写锁**   | 不支持      | `getReadWriteLock()` 支持  |
| **联锁/红锁** | 不支持      | `MultiLock` 支持多节点加锁      |


**一句话总结**：setnx 只是一个原语，Redisson 是一个完整的分布式锁框架，解决了 setnx 在生产环境中所有已知的坑。

## 20.1 Redisson 分布式锁的使用和底层原理

### 怎么用

```java
RLock lock = redissonClient.getLock("seckill:lock:" + voucherId);
try {
    // 尝试加锁，最多等待 1 秒，锁自动过期 10 秒
    boolean isLock = lock.tryLock(1, 10, TimeUnit.SECONDS);
    if (!isLock) return error("请求太频繁");
    // 业务逻辑...
} finally {
    lock.unlock();
}
```

### 底层原理（Lua 脚本保原子性）

**加锁**：

```lua
-- key不存在则设置（hset），并设置过期时间
if redis.call('exists', KEYS[1]) == 0 then
    redis.call('hset', KEYS[1], ARGV[2], 1)
    redis.call('pexpire', KEYS[1], ARGV[1])
    return nil
end
-- 是同一个线程则重入（计数+1，支持可重入锁）
if redis.call('hexists', KEYS[1], ARGV[2]) == 1 then
    redis.call('hincrby', KEYS[1], ARGV[2], 1)
    redis.call('pexpire', KEYS[1], ARGV[1])
    return nil
end
return redis.call('pttl', KEYS[1])
```

**解锁**：也是 Lua 脚本，计数减 1，减到 0 才真正删除 key。

### 看门狗机制（自动续期）

```
加锁成功
    ↓
后台启动一个 WatchDog 定时任务（默认每 10 秒执行一次）
    ↓
检查锁是否还被当前线程持有
    ↓ 是
重置过期时间为 30 秒（默认 lockWatchdogTimeout）
    ↓
业务执行完毕 → 主动解锁 → WatchDog 停止
```

**注意**：只有**不指定超时时间**的 `lock()` 才会启动看门狗；`tryLock(waitTime, leaseTime, unit)` 指定了 `leaseTime` 则不会续期，到期自动释放。

## 21. 大模型服务超时或不可用怎么处理

**① 超时控制**

- 设置合理的请求超时（如 5s），避免线程长时间阻塞
- 使用 `CompletableFuture` + 超时中断

**② 熔断降级（Sentinel/Hystrix）**

- 大模型接口错误率超过阈值，触发熔断，快速失败
- 降级响应：返回预设的兜底话术，如"智能客服暂时不可用，请稍后再试或联系人工客服"

**③ 重试机制**

- 超时后自动重试 1~2 次（注意幂等）
- 使用指数退避，避免雪崩

**④ 异步化**

- 用户发送问题后立即返回"处理中"，大模型响应后通过 WebSocket/轮询推送结果
- 避免同步等待导致请求超时

**⑤ 多模型兜底**

- 主模型不可用时，切换到备用模型或规则引擎

## 22. 秒杀结束后 Redis 库存同步回 MySQL，部分失败怎么办

**首先纠正一个认知**：秒杀中扣减库存的"最终数据"其实一直在 MySQL 中（每次异步下单都会写 DB），Redis 存的是预热的库存副本，用来快速拦截请求，不需要"秒杀结束后同步回"。

但如果面试官问的是数据一致性保障，回答思路：

**① 基于消息队列的最终一致性**

- 库存变更产生消息，消费者写 DB
- 消费失败自动重试，超过次数进死信队列人工处理

**② 定时对账任务**

- 定时任务比对 Redis 和 MySQL 的库存差异
- 发现不一致则以 MySQL 为准修正

**③ 事务消息（RocketMQ）**

- 保证"扣 Redis + 发消息"和"写 DB"要么都成功，要么都回滚

## 23. 消息队列生产消费流程 + 消息不丢失不重复消费 + Redis 实现 MQ 的缺陷

### 以 RabbitMQ 为例的生产消费流程

```
生产者 → Exchange（交换机）→ Queue（队列）→ 消费者
```

- **生产者确认**：开启 `publisher-confirm`，Broker 收到消息后回调确认
- **消息持久化**：Queue 和 Message 都设置持久化，宕机不丢
- **消费者 ACK**：处理成功才 `basicAck`，失败则 `basicNack` 重新入队

### 如何保证消息不丢失


| 环节          | 方案                  |
| ----------- | ------------------- |
| 生产者到 Broker | 开启生产者确认（Confirm 机制） |
| Broker 存储   | 队列和消息持久化到磁盘         |
| Broker 到消费者 | 手动 ACK，处理完再确认       |


### 如何保证不重复消费（幂等）

- 消息携带唯一 ID（订单 ID）
- 消费前查 DB / Redis 判断是否已处理
- DB 唯一索引兜底，重复插入直接报错捕获

### Redis 实现消息队列的缺陷


| 方式                    | 缺陷                                                         |
| --------------------- | ---------------------------------------------------------- |
| **List**（lpush/brpop） | 不支持消息确认（ACK），消费失败消息丢失；不支持多消费者组                             |
| **PubSub**            | 消息不持久化，消费者离线期间的消息全丢；不支持消息回溯                                |
| **Stream**（最完善）       | 支持 ACK 和消费者组，但功能仍比 RabbitMQ/Kafka 弱，缺乏死信队列、延迟消息、事务消息等企业级特性 |


**总结**：Redis 消息队列缺乏完善的**死信处理、延迟消息、事务消息、消息轨迹**等能力，适合轻量场景，不适合对可靠性要求极高的生产环境。

## 24. Redis 主从存在什么问题


| 问题                  | 说明                                    |
| ------------------- | ------------------------------------- |
| **数据丢失（异步复制）**      | 主节点写入成功，还未同步到从节点就宕机，这部分数据永久丢失         |
| **脑裂（Split-Brain）** | 网络分区时，旧主和新主同时接受写入，分区恢复后数据冲突，旧主数据被覆盖丢失 |
| **故障转移期间不可写**       | 主节点宕机到哨兵完成选主，中间有数秒不可用窗口               |
| **从节点只读**           | 从节点不能写，所有写压力集中在主节点，写能力无法横向扩展          |
| **读到旧数据**           | 主从复制有延迟，从节点可能读到未同步的旧数据（复制滞后）          |


**脑裂的缓解方案**：

```
# redis.conf 配置：至少有 1 个从节点正常同步，主节点才接受写入
min-slaves-to-write 1
min-slaves-max-lag 10   # 从节点延迟不超过 10 秒
```

这样主节点与从节点断开超过 10 秒后会拒绝写入，避免脑裂时数据大量写入旧主。
