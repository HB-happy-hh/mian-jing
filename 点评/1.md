# 面试题整理（点评/秒杀项目）

---

## 1. 项目中遇到的最大技术挑战是什么？最终是如何解决的？

**答题框架（STAR法则）：**

- **Situation**：秒杀场景下，瞬时流量是平时的数十倍，数据库直接承压，接口频繁超时甚至宕机。
- **Task**：需要在保证数据一致性（不超卖）的前提下，支撑高并发请求。
- **Action**：
  1. 引入 **两级缓存**（本地缓存 Caffeine + Redis），拦截绝大多数读请求；
  2. 使用 **Redis + Lua 脚本** 原子性扣减库存，避免超卖；
  3. 引入 **消息队列（RocketMQ）** 异步下单，削峰填谷；
  4. 使用 **布隆过滤器** 拦截无效请求，防止缓存穿透。
- **Result**：数据库 QPS 降低约 90%，接口 P99 延迟从 2s+ 降至 200ms 以内。

## 2. 布隆过滤器

**问：布隆过滤器用来干啥的？**

用于快速判断某个元素**是否存在于一个大集合中**，常用于防止缓存穿透（无效请求直接被拦截，不查数据库）。

**问：底层怎么实现的？**

由**位数组（bit array）+ 多个哈希函数**组成：
- 写入元素时，经过 k 个哈希函数分别映射到位数组的 k 个位置，将这些位置置为 1；
- 查询时，同样经过 k 个哈希函数，若所有位置均为 1，则**可能存在**；若有任一位置为 0，则**一定不存在**。

**问：有什么缺点？**

- **假阳性（False Positive）**：可能误判元素存在（实际不存在）；
- **不支持删除**：位是共享的，删一个元素会影响其他元素；
- **不支持扩容**：数组大小固定，需要提前估算容量。

**问：支不支持删除？不支持删除有什么别的方案？**

标准布隆过滤器**不支持删除**。替代方案：
- **计数型布隆过滤器（Counting Bloom Filter）**：将每个 bit 位改为计数器，支持删除（减计数），但占用空间更大。
- **定期重建**：全量重新加载数据到新的过滤器，替换旧的。

**问：缺点是假阳性还是假阴性？为什么？**

是**假阳性**。原因：不同元素的哈希值可能映射到相同位置（哈希碰撞），导致"以为存在但实际不存在"。而如果某位置为 0，说明没有任何元素被哈希到此，则元素一定不存在，不会有假阴性。

**问：秒杀场景里，布隆过滤器除了假阳性还有其他缺点吗？**

- **数据实时性差**：新增的商品 ID 需要手动更新过滤器，有延迟；
- **不支持查询具体值**，只能判断存在性；
- 大量写入时，哈希计算有一定 CPU 开销；
- 容量固定，无法动态扩展。

**问：用过计数 Bloom Filter 吗？**

了解过。Counting Bloom Filter 把每个 bit 替换为小整数计数器（通常 4 bit），插入时加 1，删除时减 1。代价是内存占用是普通布隆过滤器的数倍。Redis 的 `RedisBloom` 模块支持此功能。

**问：如果要更新商户信息，布隆过滤器怎么更新？**

- **商户 ID 不变**：过滤器无需更新，只需刷新 Redis 中对应的缓存值；
- **商户下线/ID 变更**：普通布隆过滤器无法删除，可选方案：
  1. 使用计数型 Bloom Filter，直接减计数；
  2. 定时全量重建过滤器（如每日低峰期）；
  3. 引入版本号或双 filter 滚动更新。

## 3. ThreadLocal

**问：ThreadLocal 里面存了哪些信息？**

在项目中，ThreadLocal 通常用于存储**当前登录用户信息**（如 userId、token 等），在拦截器中写入，在 Service 层取用，避免层层传参。

**问：有什么问题？**

- **内存泄漏**：线程池中线程复用，若未手动 `remove()`，旧的 value 会一直驻留；
- **父子线程不共享**：子线程无法继承父线程的 ThreadLocal 值（可用 `InheritableThreadLocal` 解决）；
- **异步场景失效**：使用线程池时，ThreadLocal 在子线程中为空。

**问：底层怎么实现的？**

每个 Thread 对象内部持有一个 `ThreadLocalMap`，key 是 `ThreadLocal` 实例（弱引用），value 是存储的值（强引用）。调用 `get()`/`set()` 时，以当前 ThreadLocal 实例为 key 在该 Map 中查找/写入。

**问：为什么 JVM 的 GC 不能回收 value？**

- ThreadLocalMap 的 key 是 ThreadLocal 的**弱引用**，GC 时会被回收，key 变为 null；
- 但 **value 是强引用**，只要线程还活着（线程池中线程长期存活），Entry 中的 value 就不会被 GC 回收，造成内存泄漏。

**问：GC 不能回收该怎么办？**

在使用完 ThreadLocal 后，**必须手动调用 `remove()`**，通常在拦截器的 `afterCompletion` 或 `finally` 块中执行：
```java
try {
    // 业务逻辑
} finally {
    UserHolder.removeUser();
}
```

## 4. 查询附近店铺（基于经纬度）

**问：用户在天安门，如何查附近有哪些店？**

使用 **Redis GEO** 数据结构实现（底层是 ZSet + GeoHash 编码）：

1. **数据写入**：商户上线时，将经纬度写入 Redis：
   ```
   GEOADD shops:city:北京 116.397 39.909 "shopId:123"
   ```
2. **查询附近**：
   ```
   GEORADIUS shops:city:北京 116.397 39.909 5 km ASC COUNT 20
   ```
   或使用 `GEOSEARCH`（Redis 6.2+）按距离排序返回指定范围内的商户。

3. **原理**：GeoHash 将二维经纬度编码为一维字符串，相近位置的编码具有相同前缀，存入 ZSet 的 score 中，可利用有序集合的范围查询快速检索。

**补充**：也可以在 MySQL 中用 `ST_Distance_Sphere` 计算地理距离，但在高并发下性能远不如 Redis GEO。

## 5. 流量增加 100 倍，如何重构架构？

**当前架构的瓶颈点分析：**

| 层次 | 可能出问题的地方 |
|------|----------------|
| 接入层 | 单机 Nginx 达到连接上限 |
| 应用层 | 线程池耗尽，接口超时 |
| 缓存层 | Redis 单节点 QPS 上限（约 10w/s） |
| 数据库层 | MySQL 写入成为最终瓶颈 |

**重构方案：**

1. **接入层**：Nginx 集群 + LVS/F5 负载均衡；
2. **应用层**：水平扩展服务节点，引入限流（Sentinel）+ 熔断；
3. **缓存层**：Redis 切换为**集群模式**，本地缓存（Caffeine）拦截热点数据；
4. **库存层**：Redis 预扣库存 + Lua 原子操作，数据库仅做最终落库；
5. **异步层**：MQ 异步下单，削峰填谷；
6. **数据库层**：分库分表，读写分离；
7. **防刷层**：网关层限流 + 用户维度令牌桶。

## 6. Redis + Lua 脚本防超卖，Lua 脚本怎么写？

**核心思路**：将"判断库存 + 扣减库存"做成原子操作，防止并发竞争。

```lua
-- KEYS[1]: 库存 key，如 "seckill:stock:活动ID:商品ID"
-- ARGV[1]: 购买数量

local stock = tonumber(redis.call('get', KEYS[1]))
if stock == nil then
    return -1  -- key 不存在
end
if stock < tonumber(ARGV[1]) then
    return 0   -- 库存不足
end
redis.call('decrby', KEYS[1], ARGV[1])
return 1       -- 扣减成功
```

Java 侧调用：
```java
Long result = redisTemplate.execute(script, 
    Collections.singletonList(stockKey), 
    String.valueOf(count));
```

## 7. 集群环境下，本地缓存如何保证数据一致性？

**问题**：多个节点各自维护本地缓存（Caffeine），某节点更新数据后，其他节点的本地缓存仍是旧数据。

**解决方案：发布/订阅（Pub/Sub）广播失效通知**

1. 数据更新时，先更新数据库，再删除 Redis 缓存；
2. 同时向 Redis 的某个 Channel（如 `cache:invalidate`）发布消息，携带失效的 key；
3. **每个节点订阅该 Channel**，收到消息后删除本地缓存中对应的 key；
4. 下次请求到来时，本地缓存 miss，回源 Redis 重新加载。

```java
// 发布
redisTemplate.convertAndSend("cache:invalidate", key);

// 订阅（每个节点启动时注册）
@Bean
MessageListenerAdapter listenerAdapter() {
    return new MessageListenerAdapter(cacheInvalidateListener);
}
```

## 8. 缓存删除失败，消息队列重试次数是多少？超过次数后如何处理？

**重试次数**：一般设置为 **3~5 次**，结合指数退避（如 1s、2s、4s 间隔）。

**超过最大次数后的处理策略**：

1. **写入死信队列（DLQ）**：将消息转入死信队列，人工介入排查；
2. **告警通知**：触发告警（钉钉/邮件），通知运维处理；
3. **降级兜底**：设置 key 的极短 TTL（如 1s），让其自然过期，牺牲极短时间的一致性换取可用性；
4. **记录日志**：记录失败的 key 和时间，便于后续对账修复。

## 9. 本地缓存和 Redis 协同工作的设计

**读取流程（两级缓存）：**

```
请求 → 查本地缓存（Caffeine）→ 命中直接返回
                           ↓ 未命中
              查 Redis → 命中写入本地缓存后返回
                       ↓ 未命中
              查数据库 → 写入 Redis + 本地缓存后返回
```

**优先使用本地缓存的场景**：
- 读多写少的静态数据（商品基本信息、活动配置）；
- 对一致性要求不高、可接受秒级延迟的数据；
- Redis 出现网络抖动或故障时，本地缓存作为降级兜底。

**一致性保证**：
- 更新时先写数据库，再删 Redis，再通过 Pub/Sub 广播删除各节点本地缓存；
- 本地缓存设置较短的 TTL（如 30s~60s），防止长时间不一致；
- 采用**旁路缓存（Cache-Aside）模式**，由应用层控制缓存的读写。

## 10. 缓存是怎么设计的？key/value 存了什么？

**商品/活动缓存：**

| 类型 | Key | Value |
|------|-----|-------|
| 商品信息 | `shop:{shopId}` | 商户详情 JSON（名称、地址、营业时间等） |
| 秒杀活动 | `seckill:voucher:{voucherId}` | 优惠券信息（面值、有效期、库存） |
| 库存 | `seckill:stock:{voucherId}` | 剩余库存数量（Integer） |
| 用户购买记录 | `seckill:order:{voucherId}:{userId}` | 1（标记已购买，防止重复下单） |
| 地理位置 | `shops:geo:{typeId}` | GeoHash ZSet（shopId + 经纬度） |

**静态信息**（如商户基础信息）也存入缓存，TTL 设置较长（如 30min），适合预热。

## 11. 数据库里面的主键是怎么做的？

使用**雪花算法（Snowflake）生成分布式全局唯一 ID** 作为主键：

- 64 位 long 型，由`时间戳 + 机器ID + 序列号`组成；
- 趋势递增，对 B+ 树索引友好，避免页分裂；
- 不依赖数据库自增，适合分布式/分库分表场景；
- 项目中可使用 Hutool 的 `IdUtil.getSnowflakeNextId()` 或自定义实现。

**不用 UUID 的原因**：UUID 随机性强，会造成 B+ 树频繁页分裂，写入性能差。

## 12. 如何防止有人拿程序恶意刷优惠券？

**多层防御策略：**

1. **一人一单**：Redis 中记录 `seckill:order:{voucherId}:{userId}`，Lua 脚本原子性判断是否已购买；
2. **接口限流**：用户维度令牌桶限流（Sentinel / Redis + Lua），如每秒最多 1 次请求；
3. **风控校验**：判断用户账号年龄、是否实名、历史行为特征，机器人账号命中即拦截；
4. **验证码/人机验证**：秒杀开始前要求滑块验证，防止脚本直接调接口；
5. **IP 限流**：同一 IP 短时间内请求过多则封禁；
6. **请求签名**：对关键参数加签，防止接口被直接构造请求调用；
7. **黑名单**：对已识别的恶意用户/IP 加入黑名单，网关层直接拦截。

## 13. 缓存穿透、缓存击穿、缓存雪崩

| 问题 | 定义 | 解决方案 |
|------|------|---------|
| **缓存穿透** | 查询一个**根本不存在**的数据，缓存和数据库都没有，每次都打到 DB | ① 布隆过滤器拦截无效 key；② 缓存空值（null），TTL 设置短一点（如 2min） |
| **缓存击穿** | 某个**热点 key 过期**瞬间，大量请求同时打到数据库 | ① 互斥锁（同一时间只有一个线程重建缓存）；② 逻辑过期（不设 TTL，由程序判断是否过期并异步更新） |
| **缓存雪崩** | **大量 key 同时过期**，或 Redis 宕机，导致所有请求涌入数据库 | ① 过期时间加随机值，避免同时失效；② Redis 高可用（哨兵/集群）；③ 限流 + 熔断降级；④ 本地缓存兜底 |

## 14. Redis 集群模式下 Lua 脚本涉及多 key 操作有什么问题？

**问题**：Redis 集群按 key 的 hash slot 将数据分布在不同节点，**多个 key 可能在不同节点上**。Lua 脚本要求所有操作的 key 必须在**同一个节点**，否则会报错：
```
CROSSSLOT Keys in request don't hash to the same slot
```

**解决方案：**

1. **Hash Tag**：用 `{}` 强制指定参与 hash 的部分，使相关 key 落到同一 slot：
   ```
   seckill:{voucherId}:stock
   seckill:{voucherId}:order
   ```
   两个 key 的 `{voucherId}` 部分相同，会落到同一 slot。

2. **减少 Lua 脚本中的跨 key 操作**：将逻辑拆分为单 key 操作，通过业务逻辑串联；

3. **使用 Redis 事务（MULTI/EXEC）代替**（但无法保证原子性，一般不推荐）。

## 15. 秒杀 8 点开始，设置缓存存活时间包含这个时间点能解决什么问题？有什么缺点？

**能解决的问题——缓存击穿（热点 key 提前失效）：**

在活动开始前就将库存、活动信息预热到 Redis，并设置 TTL 使其覆盖整个活动时间段（如活动时长 + 一定冗余）。这样在 8 点流量洪峰来临时，缓存必然命中，不会因 key 不存在而穿透到数据库。

**缺点：**

1. **缓存与数据库可能不一致**：若活动提前下架或库存提前修改，缓存未及时失效，会出现数据不一致；
2. **占用内存**：大量活动提前预热，Redis 内存占用提升；
3. **时钟精度问题**：TTL 是相对时间，服务重启或时钟偏移可能导致提前失效；
4. **未购买资格的用户可以提前看到库存信息**（信息泄露风险，取决于业务需要）。

## 16. 削峰解耦后系统吞吐量提升了多少？具体的压测数据？

**回答框架（没有真实数据时的参考表述）：**

引入 RocketMQ 异步下单后，接口从**同步等待数据库写入**变为**异步投递消息即返回**，响应时间从原来的 500ms+ 降至 50ms 以内。

压测数据（Jmeter，500并发，持续60s）：

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| QPS | ~800 | ~5000 |
| P99 延迟 | 2000ms | 180ms |
| 错误率 | 15%+ | <0.1% |
| 数据库 TPS | 800 | 80（消费端平滑写入） |

> 注：具体数字根据机器配置不同会有差异，压测时需说明测试环境（如：4核8G云服务器，Redis单机，MySQL单机）。
