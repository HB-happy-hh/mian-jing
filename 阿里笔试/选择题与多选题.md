### 1. HTTP/2 相比 HTTP/1.1 的关键改进点

**题目：** 关于 HTTP/2 相比 HTTP/1.1 的关键改进点，下列说法正确的是

- A. HTTP/2 强制使用 UDP 作为传输层，因此天然解决了丢包重传问题
- B. HTTP/2 通过二进制分帧与多路复用在单连接上并发多个流，但仍可能受 TCP 层丢包导致的队头阻塞影响
- C. HTTP/2 取消了头部信息，因此无需任何 header，可直接传输 body
- D. HTTP/2 只能在明文连接上工作，使用 TLS 会导致协议失效

**答案：B**

- A 错：HTTP/2 仍基于 TCP，使用 UDP 的是 HTTP/3（QUIC）
- B 正：二进制分帧+多路复用，但 TCP 层队头阻塞依然存在
- C 错：HTTP/2 使用 HPACK 压缩头部，并非取消头部
- D 错：主流浏览器只在 TLS 上支持 HTTP/2

### 2. 大模型训练中的混合精度（FP16/BF16）

**题目：** 关于大模型训练中的混合精度（如 FP16/BF16），下列说法正确的是

- A. 主要目的通常是降低显存占用并提升吞吐，且需要关注数值稳定性（如 loss scaling）
- B. 主要目的通常是提升最终精度上限，因此训练越久越好且不需要任何稳定性处理
- C. 主要目的通常是把模型参数数量减少一半，从而改变模型结构并提升泛化
- D. 主要目的是让 attention 复杂度从平方降为线性，因此对长序列最有效

**答案：A**

- A 正：降低显存 + 提升吞吐，FP16 动态范围小需 loss scaling 保证稳定
- B 错：目的是效率，不是精度上限；不做稳定处理会导致训练崩溃
- C 错：不改变参数数量和模型结构，master weights 仍以 FP32 保存
- D 错：这是 Linear Attention 等技术的目标，与混合精度无关

### 3. 稀疏矩阵转置朴素算法时间复杂度

**题目：** 某开发需要为一个大规模稀疏矩阵设计存储方案，以节省空间。若一个 m×n 的矩阵有 t 个非零元素，采用三元组顺序表存储，那么在进行矩阵转置操作时，其朴素算法的时间复杂度大致为（）

- A. O(m\*n)
- B. O(t)
- C. O(m\*t)
- D. O(n\*t)

**答案：D**

朴素转置：外层枚举 n 列，内层扫描 t 个三元组 → **O(n×t)**
- A 错：O(m\*n) 是完整二维数组转置的复杂度
- B 错：O(t) 是快速转置算法的复杂度
- C 错：外层按列遍历，是 n 次而非 m 次

### 4. 二分查找最坏比较次数

**题目：** 使用二分查找在长度为 1024 的有序数组中查找一个元素，最坏情况下需要比较的次数为（1024=2^10）

- A. 9
- B. 10
- C. 11
- D. 12

**答案：C**

数组长度 n = 2^10 时，最坏比较次数 = **log₂(n) + 1 = 10 + 1 = 11**

| 比较次数 | 区间大小 |
|---------|---------|
| 第1次 | 1024→512 |
| ... | ... |
| 第10次 | 2→1 |
| 第11次 | 与最后1个元素比较 |

### 5. SQL 查询发电量前 50% 的发电机

**题目：** 某风电场 wind_turbines（turbine_id, turbine_name, model, rated_power_kw, installation_date）和 power_outputs（output_id, turbine_id, record_date, output_kwh, wind_speed_ms），查询发电量前 50% 的发电机名称

**答案：A**

```sql
SELECT turbine_name
FROM (
    SELECT turbine_name, SUM(output_kwh) AS total_output,
           NTILE(2) OVER (ORDER BY SUM(output_kwh) DESC) AS tile
    FROM wind_turbines w
    JOIN power_outputs p ON w.turbine_id = p.turbine_id
    GROUP BY turbine_name
) t
WHERE tile = 1
```

- A 正：NTILE(2) 按总发电量降序均分2组，取第1组（前50%）
- B 错：子查询计算的是单条记录均值，不是各机器总量均值
- C 错：`LIMIT 50 PERCENT` 不是合法 SQL 语法
- D 错：`TOP 50 PERCENT` 是 SQL Server 专有语法

### 6. 分页内存管理页内偏移量位数

**题目：** 某系统采用分页式内存管理，逻辑地址空间大小为 64KB，页面大小为 4KB，则逻辑地址的页内偏移量至少需要多少位来表示（）

- A. 10
- B. 12
- C. 14
- D. 16

**答案：B**

页面大小 = 4KB = 2^12 字节，页内偏移需 **12 位**

验证：逻辑空间 64KB = 2^16，页号 = 16-12 = 4 位，共 2^4 = 16 页 ✓

### 7. UDP 的特点与工程取舍

**答案：A、C、D**

- **A 正**：UDP 不保证可靠、有序、不重复，应用层若需要需自行实现
- **B 错**：网络拥塞时 UDP 包同样被大量丢弃，有效吞吐未必优于 TCP；且 UDP 无退避机制属于不公平竞争
- **C 正**：UDP 报文超过路径 MTU 会触发 IP 分片，任一分片丢失导致整个数据报失败
- **D 正**：实时场景选 UDP，在应用层结合 FEC/重传/抖动缓冲等做体验优化（如 WebRTC、游戏、直播）

### 8. 长上下文扩展的常见方法与挑战

**答案：A、B、C**

- **A 正**：KV Cache 占用与序列长度线性正相关，上下文从 4K→128K 膨胀 32 倍，显存常成瓶颈
- **B 正**：分块/稀疏注意力可将 O(n²) 降为 O(n)，但可能影响长距离依赖建模质量
- **C 正**：RoPE 外推/缩放（YaRN、NTK-aware scaling 等）可扩展上下文，但稳定性需实验验证
- **D 错**：增大 batch size 会增加显存，与扩展上下文需求相反；长上下文需减小 batch size

### 9. AVL 树与红黑树的说法

**答案：A、C**

- **A 正**：AVL 树严格平衡，树高 ≈1.44log₂n，查询效率通常更高
- **B 错**：左右子树高度差≤1 是 AVL 的性质，红黑树保证最长路径≤最短路径×2
- **C 正**：红黑树通过重新着色（O(1)）代替旋转，插入删除平均旋转次数更少
- **D 错**：AVL 插入最多需要1次单旋或1次双旋（双旋=2次单旋），删除最坏 O(log n) 次

### 10. ER 模型映射为关系模型

**答案：A、B、C**

- **A 正**：1:N 联系在"多"端加外键，并加入联系属性（标准做法）
- **B 正**：M:N 联系需单独建联系表，主码由两端外键组合，可含联系属性
- **C 正**：1:1 联系可将一方主码作外键，也可合并为一张表，视参与约束而定
- **D 错**：弱实体主码必须包含强实体主码，弱实体不能独立唯一标识自身

### 11. 贪心算法

**答案：B、D**

- **A 错**："无后效性"是动态规划的核心条件，不是贪心算法

### 12. 操作系统系统调用

### 13. Java 静态初始化顺序

```java
public class InitOrderTest {
    private static InitOrderTest instance = new InitOrderTest();
    private static int a = 0;
    private static int b;
    private InitOrderTest() { a++; b++; }
    public static void main(String[] args) {
        InitOrderTest obj = InitOrderTest.getInstance();
        System.out.println("a = " + a);
        System.out.println("b = " + b);
    }
}
```

**答案：**
```
a = 0
b = 1
```

**执行顺序：**
1. 所有静态字段默认初始化：a=0, b=0
2. `instance = new InitOrderTest()` → 构造器：a=1, b=1
3. `a = 0` → 显式赋值覆盖：a=0
4. `private static int b;` → 无初始化表达式，b 保持 1

### 14. Java final 方法重写

```java
class Super {
    public final void display() { System.out.println("Super display"); }
}
class SubExample extends Super {
    public void display() { System.out.println("Sub display"); }  // 编译错误！
}
```

**答案：编译错误（Compile Error）**

`final` 修饰的方法不能被子类重写，编译器直接报错：
```
error: display() in SubExample cannot override display() in Super; overridden method is final
```

### 15. HashMap 内部实现

**正确描述：B、C、D**（A 错误）

- **A 错**：链表转红黑树需同时满足：链表长度≥8 **且** 数组容量≥64，缺一不可
- **B 正**：扰动函数 `hash = h ^ (h >>> 16)`，高16位与低16位异或，减少哈希冲突
- **C 正**：默认容量16，负载因子0.75，元素数量>12时触发扩容
- **D 正**：扩容时新容量=旧容量×2，元素按新容量重新分布（高位bit判断，无需重算hash）
