## 3. 介绍一下 Spring Boot

Spring Boot 是 Spring 框架的快速开发脚手架，核心特性：

1. **自动配置（Auto-Configuration）**：根据 classpath 中的依赖自动配置 Spring Bean，无需大量 XML 配置。
2. **起步依赖（Starter）**：将常用依赖打包成 starter，如 `spring-boot-starter-web`，简化 Maven/Gradle 配置。
3. **内嵌服务器**：内嵌 Tomcat/Jetty/Undertow，打包成可执行 JAR 直接运行。
4. **Actuator**：提供生产级监控端点，如健康检查、指标采集。
5. **约定大于配置**：遵循默认约定，减少开发者决策成本。

Spring Boot 本质上是对 Spring Framework 的封装，底层仍使用 Spring 的 IOC 容器和 AOP 机制。

## 4. 讲一下 AOP 和 IOC

### IOC（控制反转）
IOC 是一种设计原则，将对象的创建和依赖关系的管理从程序代码中转移到框架容器中。

- 传统方式：`UserService service = new UserService();`（程序主动创建）
- IOC 方式：对象由 Spring 容器创建并注入，程序只需声明依赖（`@Autowired`）
- 实现方式：**依赖注入（DI）**，包括构造器注入、Setter 注入、字段注入
- 好处：解耦合、便于测试、便于管理对象生命周期

### AOP（面向切面编程）
AOP 将横切关注点（如日志、事务、权限校验）从业务逻辑中分离出来，通过"切面"统一处理。

核心概念：
| 概念 | 说明 |
|------|------|
| Aspect（切面） | 横切关注点的模块化，包含 Advice 和 Pointcut |
| Pointcut（切点） | 定义在哪些方法上应用切面，用表达式描述 |
| Advice（通知） | 切面在切点处执行的动作（前置/后置/环绕等） |
| JoinPoint（连接点） | 程序执行的某个点，如方法调用 |
| Weaving（织入） | 将切面应用到目标对象的过程 |

常用场景：事务管理（`@Transactional`）、日志记录、接口限流、权限校验。

### 举例说明

**没有 AOP 时**，每个业务方法都要手动写日志，重复代码散落各处：

```java
public void transfer(int money) {
    System.out.println("transfer 方法开始执行");  // 重复代码
    account -= money;
    System.out.println("transfer 方法执行结束");  // 重复代码
}

public void pay(int money) {
    System.out.println("pay 方法开始执行");        // 重复代码
    order.pay(money);
    System.out.println("pay 方法执行结束");        // 重复代码
}
```

**有 AOP 时**，把日志逻辑抽到一个切面里，业务方法保持干净：

```java
@Aspect
@Component
public class LogAspect {

    // Pointcut（切点）：拦截 service 包下所有类的所有方法
    @Pointcut("execution(* com.example.service.*.*(..))")
    public void pt() {}

    // Advice（通知）：在切点方法执行前后插入日志
    @Around("pt()")
    public Object log(ProceedingJoinPoint joinPoint) throws Throwable {
        System.out.println(joinPoint.getSignature().getName() + " 方法开始执行");
        Object result = joinPoint.proceed(); // 执行原始方法
        System.out.println(joinPoint.getSignature().getName() + " 方法执行结束");
        return result;
    }
}
```

业务方法只剩业务逻辑：

```java
public void transfer(int money) {
    account -= money;  // 只剩业务逻辑
}

public void pay(int money) {
    order.pay(money);  // 只剩业务逻辑
}
```

各概念与例子的对应关系：

| 概念 | 对应例子中的什么 |
|------|----------------|
| Aspect（切面） | `LogAspect` 这个类 |
| Pointcut（切点） | `execution(* com.example.service.*.*(..))` 表达式，匹配了 `transfer` 和 `pay` |
| Advice（通知） | `log()` 方法里的打印日志逻辑 |
| JoinPoint（连接点） | `transfer()`、`pay()` 这些可以被拦截的方法 |
| Weaving（织入） | Spring 启动时自动给业务方法套上代理、插入日志的这个过程 |

> 一句话总结：**你只管写业务，切面在你不知情的情况下，在方法执行前后偷偷插入额外逻辑。**

## 5. AOP 如何织入，静态织入和动态织入

### 静态织入（编译期织入）
在**编译阶段**将切面代码编译进字节码，生成新的 `.class` 文件。
- 代表：**AspectJ** 编译器（ajc）
- 优点：性能好，无运行时开销
- 缺点：需要特殊编译器，灵活性差

### 动态织入（运行期织入）
在**运行时**通过代理机制动态地将切面逻辑织入目标对象。Spring AOP 默认使用此方式。

两种动态代理：
1. **JDK 动态代理**：
   - 要求目标类实现接口
   - 基于 `java.lang.reflect.Proxy`，生成实现了相同接口的代理类
   - `InvocationHandler.invoke()` 中插入增强逻辑

2. **CGLIB 动态代理**：
   - 目标类无需实现接口
   - 通过继承目标类生成子类，重写方法插入增强逻辑
   - Spring Boot 2.x 后默认使用 CGLIB

Spring 选择策略：有接口 → JDK代理，无接口 → CGLIB（可通过 `proxyTargetClass=true` 强制使用CGLIB）。

## 6. 讲一下 JWT 令牌

JWT（JSON Web Token）是一种无状态的身份验证令牌，由三部分组成，用 `.` 分隔：

```
Header.Payload.Signature
```

- **Header**：算法类型（如 HS256）和令牌类型（JWT），Base64 编码
- **Payload**：声明（Claims），包含用户ID、角色、过期时间等，Base64 编码（非加密）
- **Signature**：用密钥对 `Header + Payload` 进行签名，防止篡改

工作流程：
1. 用户登录 → 服务端生成 JWT → 返回给客户端
2. 客户端将 JWT 存储在 LocalStorage 或 Cookie
3. 每次请求在 `Authorization: Bearer <token>` 头中携带
4. 服务端验证签名合法性和过期时间，无需查询数据库

优点：无状态、跨域友好、服务端无需存储 Session  
缺点：Token 无法主动失效（除非引入黑名单），Payload 信息可见（敏感信息需加密）

---

## 7. JWT 和 Cookie、Session 的关系

| 对比项 | Session+Cookie | JWT |
|--------|----------------|-----|
| 存储位置 | Session 存服务端，SessionId 存 Cookie | Token 存客户端（Cookie/LocalStorage） |
| 服务端状态 | 有状态（需存储Session） | 无状态（服务端不存储） |
| 扩展性 | 分布式需共享Session（如Redis） | 天然支持分布式，任意节点可验证 |
| 安全性 | CSRF 风险（Cookie自动携带） | XSS 风险（LocalStorage），CSRF 较小 |
| 失效控制 | 服务端可直接销毁Session | 需黑名单机制才能主动失效 |

关系说明：
- JWT 可以存放在 Cookie 中传输，但本质上是替代 Session 的认证机制
- Cookie 只是传输 JWT 的载体之一，JWT 也可放在请求头中
- Session 是服务端有状态的认证方案，JWT 是无状态方案，两者是替代关系

## 8. JWT Token 失效后如何续期

### 方案一：双Token（推荐）
- **Access Token**：有效期短（如15分钟），用于鉴权
- **Refresh Token**：有效期长（如7天），用于刷新 Access Token
- 流程：Access Token 过期 → 客户端用 Refresh Token 请求新的 Access Token → 若 Refresh Token 也过期则重新登录
- Refresh Token 存储在 Redis 中，可主动失效

### 方案二：滑动过期
每次请求时检查 Token 剩余有效期，若小于某个阈值（如5分钟），自动在响应头中返回新 Token，客户端替换本地旧 Token。

### 方案三：Token 黑名单
用户主动退出或 Token 被吊销时，将 Token 的 jti（唯一标识）存入 Redis 黑名单，每次验证时检查是否在黑名单中。

## 9. Redis + MQ 的逻辑设计

以秒杀场景为例：

### 整体流程
```
用户请求 → Redis 校验（库存/资格） → 发送MQ消息 → 返回成功
                                         ↓
                               MQ消费者 → 写入数据库
```

### 详细步骤
1. **Redis 预扣库存**：将库存加载到 Redis，用 `DECR` 原子减库存，判断是否 ≥ 0
2. **资格校验**：用 Redis Set 记录已购用户，防止重复购买（`SADD userId`，返回0则已购）
3. **以上两步用 Lua 脚本保证原子性**
4. **发送 MQ 消息**：校验通过后将订单信息发送到 RabbitMQ/Kafka
5. **异步消费**：MQ 消费者接收消息，执行真正的数据库操作（扣库存、创建订单）
6. **结果通知**：通过 WebSocket 或轮询告知用户下单结果

### 好处
- Redis 处理高并发（QPS 10万+），避免直接打数据库
- MQ 削峰填谷，数据库按自身处理能力消费
- 异步解耦，提升响应速度

## 10. Redis 数据结构设计

| 数据结构 | 底层实现 | 典型使用场景 |
|----------|----------|-------------|
| **String** | SDS | 缓存对象（JSON）、计数器、分布式锁、Token存储 |
| **Hash** | ziplist/hashtable | 存储对象字段（如用户信息），避免序列化整个对象 |
| **List** | quicklist | 消息队列、最新消息列表、关注的人的动态（Feed流拉模式） |
| **Set** | intset/hashtable | 去重（已购用户）、共同好友、抽奖池 |
| **ZSet（有序集合）** | ziplist/skiplist | 排行榜、带权重的任务队列、Feed流（时间戳作score） |
| **BitMap** | String扩展 | 用户签到、布隆过滤器、在线状态 |
| **HyperLogLog** | 统计算法 | UV统计（不精确但省内存） |
| **Geo** | ZSet扩展 | 附近商铺、地理位置搜索 |

项目中的设计示例：
- 商铺缓存：`String`，key=`shop:id`，value=JSON
- 用户签到：`BitMap`，key=`sign:userId:202604`
- 附近商铺：`Geo`，key=`shop:geo:typeId`
- 秒杀库存：`String`，key=`seckill:stock:voucherId`
- 已购用户：`Set`，key=`seckill:order:voucherId`
- 关注列表：`ZSet`，score=关注时间戳

## 11. Lua 脚本使用注意事项

Lua 脚本在 Redis 中以原子方式执行，注意事项：

1. **原子性**：Lua 脚本执行期间 Redis 不会处理其他命令，适合需要原子操作的场景（如判断+修改）
2. **禁止耗时操作**：Lua 是单线程执行，脚本内不能有耗时循环或大量计算，否则阻塞 Redis
3. **Key 必须显式传入**：所有用到的 key 必须通过 `KEYS[]` 参数传入，不能硬编码，保证集群模式下 key 哈希槽一致（集群模式要求所有 key 在同一槽）
4. **错误处理**：Lua 脚本中的错误会导致整个脚本回滚，但不影响之前已执行的命令（Redis Lua 非事务性回滚）
5. **脚本缓存**：用 `EVALSHA` 代替 `EVAL`，通过 SHA1 引用已缓存脚本，减少网络传输
6. **不要在脚本中使用随机命令**：如 `RANDOMKEY`、`SRANDMEMBER`，在主从复制时可能产生不一致
7. **版本兼容**：Redis 7.0 之前 Lua 脚本不支持 `redis.call` 的流式API，注意版本差异

## 12. 分布式锁如何实现

### 方案一：Redis 原生实现
```
SET lock:key uniqueValue NX PX 30000
```
- `NX`：不存在才设置（互斥）
- `PX 30000`：30秒超时防死锁
- **释放锁**：必须先判断是自己的锁再删除（用 Lua 脚本保证原子性）

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

存在问题：锁过期续期困难、主从切换时锁丢失

### 方案二：Redisson（推荐）
Redisson 是基于 Redis 的 Java 分布式锁框架，解决了上述问题：

- **看门狗机制**：持锁期间自动续期（默认10秒检测一次，续期到30秒），防止业务未完成锁已过期
- **可重入锁**：同一线程可多次获取同一把锁（用 Hash 记录重入次数）
- **RedLock**：向多个独立 Redis 节点申请锁，超过半数成功才视为加锁成功，解决主从切换问题

```java
RLock lock = redissonClient.getLock("lock:key");
try {
    lock.lock(); // 阻塞等待
    // 业务逻辑
} finally {
    lock.unlock();
}
```

## 13. 对数据库索引的理解

索引是数据库中用于**加速查询**的数据结构，本质是用空间换时间。

MySQL InnoDB 的索引底层使用 **B+ 树**：
- 所有数据存储在叶子节点，叶子节点通过双向链表连接（便于范围查询）
- 非叶子节点只存储索引键值，节点扇出大，树高低（通常3-4层），减少 IO 次数
- 每次查询从根节点开始，时间复杂度 O(log n)

索引分类：
- **聚簇索引**：叶子节点存储完整行数据，InnoDB 中主键即聚簇索引
- **二级索引（非聚簇）**：叶子节点存储主键值，查询时需**回表**到聚簇索引获取完整数据
- **联合索引**：多列组合，遵循最左匹配原则
- **覆盖索引**：查询所需列均在索引中，无需回表

---

## 14. 索引可以随便加吗？

不可以，索引有代价：

1. **占用存储空间**：每个索引都需要额外的磁盘空间存储 B+ 树
2. **降低写性能**：每次 INSERT/UPDATE/DELETE 都需要维护所有索引的 B+ 树结构
3. **优化器选择开销**：索引过多会增加查询优化器的选择成本
4. **索引失效风险**：不合理的索引可能被优化器忽略

应该建索引的场景：
- 频繁出现在 WHERE、JOIN ON、ORDER BY、GROUP BY 中的字段
- 区分度高的字段（如用户ID），不适合低区分度字段（如性别）
- 常用的查询组合适合建联合索引

不适合建索引：
- 数据量极小的表
- 频繁更新的字段
- 区分度低的字段
- 不参与查询条件的字段

## 15. 项目中的索引设计

以点评项目为例：

- `tb_user` 表：`phone` 字段建**唯一索引**（登录时按手机号查询，且手机号唯一）

- `tb_voucher_order` 表：建 `(user_id, voucher_id)` **联合索引**（一人一单查询：`WHERE user_id=? AND voucher_id=?`）
- `tb_blog` 表：`user_id` 建**普通索引**（查询某用户的所有博客）
- `tb_follow` 表：`(user_id, follow_user_id)` 建**联合唯一索引**（防重复关注，查询关注关系）

## 16. 为什么不对其他字段加索引？

以 `tb_shop` 的 `name` 字段为例不加索引：
- 按名称查询频率低，主要通过分类/区域过滤
- 模糊查询（`LIKE '%关键词%'`）无法使用 B+ 树索引
- 维护索引有写性能代价，收益不高

通用原则：**不参与查询条件、区分度低、写多读少的字段不适合加索引**。若确实需要全文搜索，应考虑 Elasticsearch 而非数据库索引。

## 17. 讲一下覆盖索引

覆盖索引指查询所需的所有列都包含在索引中，**无需回表**，直接从索引中获取数据。

示例：
```sql
-- 联合索引 (user_id, voucher_id)
-- 覆盖索引：只查这两个字段，无需回表
SELECT user_id, voucher_id FROM tb_voucher_order WHERE user_id = 1;

-- 非覆盖索引：需要查 status 字段，需回表
SELECT user_id, voucher_id, status FROM tb_voucher_order WHERE user_id = 1;
```

优势：
- 减少 IO 次数（不需要读取主键索引的完整行数据）
- 对大表查询性能提升明显

使用 `EXPLAIN` 查看执行计划，`Extra` 列出现 `Using index` 表示使用了覆盖索引。

---

## 18. 讲一下最左匹配原则

联合索引 `(a, b, c)` 按照最左优先匹配：

- `WHERE a=1` ✅ 使用索引（匹配a）
- `WHERE a=1 AND b=2` ✅ 使用索引（匹配a, b）
- `WHERE a=1 AND b=2 AND c=3` ✅ 使用索引（匹配a, b, c）
- `WHERE b=2` ❌ 不使用索引（跳过了a）
- `WHERE a=1 AND c=3` ✅ 部分使用索引（只匹配a，c无法使用）
- `WHERE a=1 AND b>2 AND c=3` ✅ 匹配a, b（b是范围查询，c无法使用）

原因：B+ 树联合索引是按 (a, b, c) 顺序排序的，没有 a 的前提下无法在 b 上定位，因为 b 的顺序是在相同 a 值下才有序。

MySQL 8.0 引入**索引跳跃扫描（Index Skip Scan）**，部分场景可跳过最左列，但性能不如完整匹配。

---

## 19. 没有违反最左匹配但依然不走索引的情况

1. **索引列上使用函数**：`WHERE YEAR(create_time) = 2024`，函数导致索引失效
2. **隐式类型转换**：字段是 `VARCHAR`，查询用数字 `WHERE phone = 13800138000`，MySQL 会转换类型导致索引失效
3. **使用 != 或 <>**：不等于操作符通常不走索引
4. **LIKE 以通配符开头**：`LIKE '%abc'` 不走索引，`LIKE 'abc%'` 走索引
5. **OR 条件有非索引字段**：`WHERE id=1 OR name='xx'`，若 name 无索引则全表扫描
6. **数据量极小**：优化器判断全表扫描更快（回表成本 > 全表）
7. **索引区分度太低**：如性别字段，优化器选择全表扫描

---

## 20. 慢查询如何处理

### 发现慢查询
- 开启慢查询日志：`slow_query_log=1`，`long_query_time=2`（超过2秒记录）
- 使用 `mysqldumpslow` 分析慢查询日志
- 使用 `SHOW PROCESSLIST` 查看当前执行的SQL

### 分析慢查询
使用 `EXPLAIN` 查看执行计划，关注：
- `type`：ALL（全表扫描）→ index → range → ref → eq_ref → const（效率递增）
- `key`：实际使用的索引
- `rows`：估计扫描行数
- `Extra`：是否有 `Using filesort`（文件排序）、`Using temporary`（临时表）

### 优化手段
1. **加索引**：对查询条件字段建合适的索引
2. **优化 SQL**：避免 SELECT *，减少子查询，改用 JOIN
3. **优化索引设计**：利用覆盖索引避免回表
4. **分页优化**：深分页 `LIMIT 10000,10` 改为游标分页
5. **读写分离**：主库写，从库读，分散查询压力
6. **分库分表**：数据量过大时进行水平拆分

## 21. TCP 协议及如何保证数据安全

TCP（传输控制协议）是面向连接的、可靠的、基于字节流的传输层协议。

保证数据安全的机制：
1. **序列号与确认应答（ACK）**：每个数据包有序列号，接收方发送 ACK 确认，未收到 ACK 则重传
2. **超时重传**：发送方设置定时器，超时未收到 ACK 则重发数据包
3. **校验和**：TCP 头部包含校验和，接收方验证数据完整性，不一致则丢弃并请求重传
4. **流量控制（滑动窗口）**：接收方通过窗口大小告知发送方能接收多少数据，防止接收缓冲区溢出
5. **拥塞控制**：慢启动、拥塞避免、快速重传、快速恢复，避免网络拥塞
6. **有序传输**：接收方按序列号重新排列乱序到达的数据包

## 22. TCP 和 UDP 的相同点和不同点

### 相同点
- 都是传输层协议
- 都使用端口号区分应用程序
- 都可以进行数据传输

### 不同点

| 对比项 | TCP | UDP |
|--------|-----|-----|
| 连接 | 面向连接（三次握手） | 无连接 |
| 可靠性 | 可靠（确认、重传、有序） | 不可靠（尽力而为） |
| 速度 | 较慢 | 快 |
| 头部开销 | 20字节 | 8字节 |
| 流量控制 | 有（滑动窗口） | 无 |
| 拥塞控制 | 有 | 无 |
| 适用场景 | HTTP、FTP、邮件（数据准确性优先） | 视频直播、DNS、游戏（实时性优先） |

## 23. TCP 如何建立连接（三次握手）

```
客户端                    服务端
  |  --SYN(seq=x)-->       |   第一次握手：客户端发送SYN，进入SYN_SENT状态
  |  <--SYN+ACK(seq=y,ack=x+1)--  第二次握手：服务端回SYN+ACK，进入SYN_RCVD
  |  --ACK(ack=y+1)-->     |   第三次握手：客户端发送ACK，双方进入ESTABLISHED
```

**为什么需要三次握手？**
- 确认双方的发送和接收能力都正常
- 防止已失效的连接请求报文突然到达服务端造成错误（历史连接问题）
- 两次握手不够：服务端无法确认客户端能收到自己的数据

**四次挥手（关闭连接）**：
- 客户端发 FIN → 服务端 ACK → 服务端发 FIN → 客户端 ACK → 等待 2MSL → 关闭
- 四次而非三次：TCP 是全双工，两个方向需要分别关闭

## 24. 线程和进程的区别

| 对比项 | 进程 | 线程 |
|--------|------|------|
| 定义 | 程序的一次执行实例，资源分配的基本单位 | 进程内的执行单元，CPU调度的基本单位 |
| 资源 | 独立的内存空间（代码段、数据段、堆栈） | 共享进程的内存，有独立的栈和程序计数器 |
| 通信 | 复杂（IPC：管道、消息队列、共享内存等） | 简单（共享内存，但需同步）|
| 开销 | 创建/切换开销大 | 创建/切换开销小 |
| 安全性 | 进程间隔离，一个崩溃不影响其他进程 | 线程共享进程资源，一个线程崩溃可能影响整个进程 |
| 并发 | 多进程并发 | 多线程并发（更轻量） |

## 25. 进程间通信方式（IPC）

1. **管道（Pipe）**：单向、半双工，只能在父子进程间使用；命名管道（FIFO）可在无关进程间使用
2. **消息队列**：内核维护的消息链表，进程通过 msgsnd/msgrcv 收发消息
3. **共享内存**：最快的IPC方式，多进程映射同一块物理内存，需配合信号量同步
4. **信号量（Semaphore）**：用于进程/线程同步，控制对共享资源的访问
5. **信号（Signal）**：用于通知进程某事件发生，如 SIGKILL、SIGTERM
6. **Socket**：最通用，可用于同机进程间和跨网络进程间通信

## 29. HTTPS 中证书的作用及使用方式

### 证书的作用
1. **身份认证**：证明服务器确实是所声称的域名所有者，防止中间人攻击（假冒服务器）
2. **公钥分发**：证书中包含服务器的公钥，客户端用于加密密钥协商
3. **信任链**：证书由受信任的 CA（证书颁发机构）签发，浏览器内置根证书，可验证信任链

### 证书使用流程（TLS握手）
1. 客户端发送 `ClientHello`（支持的加密套件、随机数）
2. 服务端发送 `ServerHello` + 证书
3. 客户端验证证书：检查 CA 签名、域名匹配、有效期
4. 密钥协商：生成会话密钥（对称密钥）
5. 后续通信使用会话密钥加密

---

## 30. HTTPS 使用的加密方式

HTTPS 结合了两种加密方式：

1. **非对称加密（RSA/ECDSA）**：用于 TLS 握手阶段的**身份验证和密钥协商**
   - 公钥加密，私钥解密
   - 计算开销大，不适合大量数据加密

2. **对称加密（AES/ChaCha20）**：用于握手后的**数据传输加密**
   - 同一密钥加解密
   - 计算高效，适合大量数据

3. **哈希算法（SHA-256等）**：用于消息认证码（MAC），验证数据完整性

---

## 31. HTTPS 什么时候用对称加密，什么时候用非对称加密

- **TLS 握手阶段（非对称加密）**：
  - 客户端用服务器公钥加密预主密钥（Pre-Master Secret）发送给服务器
  - 服务器用私钥解密得到预主密钥
  - 双方基于预主密钥和随机数生成对称会话密钥
  - 目的：安全地交换对称密钥（非对称加密保证密钥传输安全）

- **数据传输阶段（对称加密）**：
  - 握手完成后，使用协商好的对称会话密钥加密所有数据
  - 原因：对称加密速度快（比非对称加密快100-1000倍），适合持续的数据传输

简记：**非对称加密保护密钥交换，对称加密保护数据传输**。

## 35. 讲一下 volatile 关键字

`volatile` 是 Java 中的关键字，用于修饰变量，有两个作用：

### 1. 保证可见性
- 每次读取 volatile 变量时，从主内存读，而不是 CPU 缓存
- 每次写入 volatile 变量时，立即刷新到主内存，并使其他 CPU 缓存中的该变量失效
- 解决了 JMM（Java 内存模型）中的缓存可见性问题

### 2. 禁止指令重排序
- 通过**内存屏障**防止编译器和 CPU 对 volatile 变量前后的指令进行重排
- 单例模式双重检查锁（DCL）中 `instance` 必须用 volatile，防止未初始化完成就被其他线程看到

```java
private volatile static Singleton instance;
```

---

## 36. volatile 可以保证原子性吗？

**不能**。volatile 只保证可见性和有序性，不保证原子性。

反例：
```java
volatile int count = 0;
count++; // 非原子操作，等价于：read → add → write（三步）
```
即使 `count` 是 volatile，多线程执行 `count++` 仍然会出现数据丢失。

原因：`count++` 是复合操作，volatile 只能保证每次读写的原子性（long/double 的64位读写保证原子），但不能保证"读-改-写"复合操作的原子性。

---

## 37. 如何保证原子性

1. **synchronized 关键字**：
   ```java
   synchronized(this) { count++; }
   ```

2. **ReentrantLock**：
   ```java
   lock.lock(); try { count++; } finally { lock.unlock(); }
   ```

3. **原子类（java.util.concurrent.atomic）**：
   ```java
   AtomicInteger count = new AtomicInteger(0);
   count.incrementAndGet(); // CAS 实现，无锁原子操作
   ```
   原子类包括：`AtomicInteger`、`AtomicLong`、`AtomicReference`、`AtomicStampedReference`（解决ABA问题）

4. **LongAdder**（高并发计数推荐）：比 AtomicLong 更高效，分段累加减少竞争

## 39. 讲一下扩容，以及扩容是为了解决什么

### 扩容是为了解决什么
HashMap 通过**哈希取模**将 key 分配到数组桶中。当数据量增大时，同一个桶中的链表/红黑树会越来越长，查询时间复杂度退化，**扩容是为了维持低负载因子，保证查询 O(1) 的平均时间复杂度**。

### HashMap 扩容机制（JDK 8）
- 默认初始容量：**16**，负载因子：**0.75**
- 触发扩容条件：元素数量 > `capacity × loadFactor`（即超过12个触发）
- 扩容大小：**2倍**（保证容量始终是2的幂次，方便用位运算取模：`hash & (capacity-1)`）
- 扩容过程：创建新数组，将旧数组中每个元素重新哈希分配（**rehash**）
- 优化：JDK 8 中，元素在新数组中的位置只有两种可能：原位置 或 原位置+旧容量，不需要重新计算hash

---

## 40. 解决哈希冲突的方式

1. **链地址法（拉链法）**：同一桶内用链表存储冲突元素（HashMap 使用）
   - JDK 8 优化：链表长度 ≥ 8 且数组长度 ≥ 64 时，链表转为**红黑树**（O(n) → O(logn)）
   - 红黑树长度 ≤ 6 时退化回链表

2. **开放地址法**：冲突时寻找下一个空位
   - 线性探测：依次向后找空位（聚集问题）
   - 二次探测：按 1, 4, 9... 步长探测
   - 双重哈希：用第二个哈希函数决定步长

3. **再哈希法**：冲突时用另一个哈希函数重新计算

4. **公共溢出区**：将冲突元素存入单独的溢出表

## 42. 讲一下 ConcurrentHashMap

ConcurrentHashMap 是线程安全的高性能哈希表，是 Hashtable 的替代品。

### JDK 7 实现（分段锁）
将数组分成若干 **Segment**（默认16个），每个 Segment 继承 ReentrantLock。不同 Segment 的操作互不干扰，最多支持16个线程并发写。

### JDK 8 实现（CAS + synchronized）
- 放弃 Segment，改为对**每个桶（链表头节点/红黑树根节点）**加 synchronized 锁
- 插入时：桶为空 → CAS 无锁插入；桶不为空 → synchronized 锁住头节点后插入
- 读操作：volatile 保证可见性，完全无锁
- 并发粒度更细，性能更好

### 关键机制
- `size()` 使用 `CounterCell` 数组分段计数，减少竞争
- 不允许 null key 和 null value（防止歧义：get返回null是key不存在还是value为null？）

## 43. 讲一下 CAS 锁

CAS（Compare And Swap，比较并交换）是一种**无锁并发**机制，依赖 CPU 原子指令（`CMPXCHG`）。

### 原理
```
CAS(内存地址, 期望值, 新值)
→ 若内存值 == 期望值，则将内存值更新为新值，返回 true
→ 若内存值 != 期望值（已被其他线程修改），返回 false，重试（自旋）
```

### 优点
- 无锁，无线程阻塞和唤醒的开销
- 适合竞争不激烈的场景

### 缺点
1. **ABA 问题**：值从 A→B→A，CAS 无法感知中间变化
   - 解决：`AtomicStampedReference`（带版本号的引用）
2. **自旋消耗 CPU**：竞争激烈时，失败的线程一直循环，浪费 CPU
   - 解决：引入自适应自旋或退化为锁
3. **只能保证单个变量的原子性**：多变量操作需用 `AtomicReference` 封装对象

## 44. MySQL 中都有什么锁

### 按粒度分
1. **全局锁**：`FLUSH TABLES WITH READ LOCK`，锁整个数据库，用于全库备份
2. **表级锁**：`LOCK TABLES t READ/WRITE`，开销小，并发低
3. **行级锁**（InnoDB）：并发度最高

### InnoDB 行锁类型
1. **记录锁（Record Lock）**：锁定单条记录
2. **间隙锁（Gap Lock）**：锁定索引记录间的间隙，防止幻读（在 RR 隔离级别下）
3. **临键锁（Next-Key Lock）**：记录锁 + 间隙锁，InnoDB 默认使用，锁定范围 `(a, b]`
4. **插入意向锁（Insert Intention Lock）**：INSERT 操作时在间隙中设置，多个INSERT可并发不互相阻塞

### 按意向分
- **意向共享锁（IS）/ 意向排他锁（IX）**：表级锁，在加行锁前加，告知其他事务该表有行锁存在

### 按读写分
- **共享锁（S锁/读锁）**：`SELECT ... LOCK IN SHARE MODE`，允许多个事务同时读
- **排他锁（X锁/写锁）**：`SELECT ... FOR UPDATE` / DML，独占，不允许其他锁

### 乐观锁/悲观锁（逻辑层面）
- **悲观锁**：认为冲突必然发生，加锁再操作（如 `SELECT FOR UPDATE`）
- **乐观锁**：认为冲突少，操作完提交时检查版本号（`UPDATE ... WHERE version=xxx`）

## 45. MySQL 事务的特性（ACID）

| 特性 | 说明 | InnoDB 实现 |
|------|------|------------|
| **原子性（Atomicity）** | 事务中所有操作要么全部成功，要么全部回滚 | Undo Log（记录反向操作，用于回滚）|
| **一致性（Consistency）** | 事务前后数据库保持完整性约束（如唯一性、外键） | 由原子性+隔离性+持久性共同保证 |
| **隔离性（Isolation）** | 多个事务并发执行时互不干扰 | MVCC + 锁机制 |
| **持久性（Durability）** | 事务提交后，数据永久保存，即使系统崩溃 | Redo Log（WAL，预写日志）|

## 46. MySQL 的隔离级别

SQL 标准定义了 4 种隔离级别，解决不同的并发问题：

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 说明 |
|----------|------|-----------|------|------|
| **READ UNCOMMITTED（读未提交）** | ✅可能 | ✅可能 | ✅可能 | 最低隔离，几乎不用 |
| **READ COMMITTED（读已提交）** | ❌避免 | ✅可能 | ✅可能 | Oracle默认 |
| **REPEATABLE READ（可重复读）** | ❌避免 | ❌避免 | ⚠️基本避免 | **MySQL InnoDB默认** |
| **SERIALIZABLE（串行化）** | ❌避免 | ❌避免 | ❌避免 | 最高隔离，性能最差 |

### 并发问题解释
- **脏读**：读到其他事务**未提交**的数据
- **不可重复读**：同一事务内，两次读同一行数据结果不同（其他事务 UPDATE 并提交）
- **幻读**：同一事务内，两次查询返回的行数不同（其他事务 INSERT/DELETE 并提交）

### InnoDB RR 级别下如何解决幻读
- **快照读**（普通SELECT）：通过 **MVCC**（多版本并发控制）读取历史版本，天然无幻读
- **当前读**（SELECT FOR UPDATE、DML）：通过 **Next-Key Lock** 锁定范围，防止幻读

### MVCC 实现原理
每行数据有两个隐藏列：`trx_id`（创建/修改的事务ID）和 `roll_pointer`（指向 Undo Log 中的旧版本）。事务开始时生成 **Read View**，根据 trx_id 判断哪个版本的数据对本事务可见，实现无锁的快照读。
