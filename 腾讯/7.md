1. Kafka 用来做什么的？

答：Kafka 是一个分布式消息队列 / 事件流平台，核心用途有三个：
① **异步解耦**：生产者和消费者不直接依赖，互不影响对方的性能和可用性。
② **流量削峰**：突发大流量时，消息先堆积在 Kafka，消费者按自己的节奏消费，保护下游服务不被打垮。
③ **可靠投递 + 重试**：消息持久化到磁盘，消费失败可以重新消费，保证最终一致性。
常见场景：订单系统、日志收集、实时数据管道、事件溯源等。

1. Redis+Lua 进行原子性下单，为什么还需要 Kafka？怎么证明你这么做是合理的？我现在怀疑你 Kafka 这一步没必要，怎么反驳？

答：Redis+Lua 解决的是**库存扣减的原子性**（check-then-decrement 不被并发打断），但扣库存之后还有很多重操作：写订单表、扣账户余额、发通知、触发物流……这些都是数据库 IO，不适合放在 Redis 事务里同步完成。

**为什么需要 Kafka：**

- 如果扣完库存后直接同步写 DB，一旦 DB 抖动或下游超时，整个下单链路就挂了，而库存已扣无法回滚，会出现"扣了库存没有订单"的数据不一致问题。
- 引入 Kafka 后，Redis 扣库存成功 → 投一条消息到 Kafka → 消费者异步创建订单。Kafka 消息持久化，消费失败可重试，最终订单一定会被创建，实现**最终一致性**。

**反驳"Kafka 没必要"：**

- 没有 Kafka，高并发下 DB 直接承压，QPS 会被 DB 瓶颈卡死；有 Kafka 做缓冲，DB 压力平滑可控。
- 没有 Kafka，下游任意一个服务（物流、通知）宕机，整个下单请求都失败；有 Kafka，各消费者独立，一个挂了不影响其他。
- 本质：Redis 保证**强一致的库存扣减**，Kafka 保证**可靠的后续流程执行**，两者解决的问题不同，不能互相替代。

1. Redis 的性能是多少？具体指标

答：

- **吞吐量**：单实例读写约 **10万 QPS**（简单 GET/SET），官方 benchmark 在高配服务器上可达 100万+ QPS。
- **延迟**：99% 的命令响应时间在 **1ms 以内**，P99 延迟通常 < 0.5ms。
- **内存带宽**：Redis 是纯内存操作，瓶颈在网络 I/O 和内存带宽，不在磁盘。
- **线程模型**：命令执行是**单线程**（避免并发锁开销），Redis 6.0 起网络 I/O 引入多线程，进一步提升吞吐。
- **数据结构影响**：简单 String 最快；ZSET（跳表）、HyperLogLog 等复杂结构操作会慢一些，但通常仍在微秒级。

1. 开发过程中有用到大模型吗？参考开源资料开发时，怎么确保别人的是对的？

答：有用，主要用于：生成样板代码、解释陌生 API、辅助 debug（贴错误栈让它分析）、写单元测试。

参考开源资料时确保正确性的方法：
① **读测试用例**：好的开源项目都有完善的 test，测试是最真实的使用说明。
② **自己跑一遍**：把 demo 在本地跑通，观察实际输出，不只是看代码。
③ **交叉验证**：同一个问题查多个来源（官方文档 > GitHub issue > 博客），不只依赖某一篇文章。
④ **理解原理**：不要只复制代码，要能解释这段代码为什么这样写，能解释才说明真的理解了。
⑤ **看 issue / PR**：开源项目的 issue 区往往记录了常见坑和已知 bug，是排查问题的重要资料。

1. 你觉得现在 AI 对项目开发有什么样的帮助？

答：
① **提速**：样板代码、CRUD、配置文件、正则表达式这类重复性工作，AI 可以秒级生成，节省大量时间。
② **降低学习成本**：遇到不熟悉的框架或语言，可以直接问 AI 要示例代码，比翻文档快得多。
③ **辅助 debug**：把报错信息和相关代码贴给 AI，通常能快速定位问题方向。
④ **代码审查**：让 AI 检查代码里的潜在问题、边界条件、性能瓶颈。
⑤ **文档生成**：注释、README、接口文档可以让 AI 辅助撰写。

局限：AI 对业务逻辑理解有限，生成的代码需要人工审查，不能盲目信任。

1. 如果现在让你使用大模型辅助去开发一个项目，你会怎么做？

答：分阶段使用：

**需求 / 设计阶段**：把需求描述给 AI，让它帮助梳理功能列表、提出可能遗漏的边界场景、给出技术选型建议，作为头脑风暴的辅助。

**开发阶段**：

- 让 AI 生成脚手架、数据模型、接口定义等样板代码，自己在此基础上修改。
- 遇到具体算法或 API 用法不确定时，先问 AI 给出思路，再自己实现和验证。
- 复杂逻辑自己写，简单重复的让 AI 生成。

**测试阶段**：让 AI 根据函数签名和注释生成单元测试用例，包括边界值和异常场景。

**Review 阶段**：提交 PR 前，把 diff 贴给 AI 做初步审查，检查明显问题。

核心原则：**AI 做助手，人做决策**，所有 AI 生成的代码都必须自己读懂、测过才能合入。

1. 怎么保证大模型生成的东西是正确的？如果是和其他人合作开发，你身为 leader，怎么确保其他人写的代码都是对的？

答：

**保证 AI 生成代码的正确性：**
① 自己读懂每一行，能解释逻辑才算验收通过。
② 写单元测试覆盖正常路径、边界值、异常场景。
③ 在 staging 环境跑集成测试，观察实际行为。
④ 对关键业务逻辑不依赖 AI，自己手写并 review。

**作为 leader 保证团队代码质量：**
① **Code Review 制度**：所有 PR 必须经过至少一个人 review 才能合并，review checklist 包括逻辑正确性、异常处理、性能风险、测试覆盖。
② **CI/CD 门禁**：单元测试覆盖率要求（如 ≥ 80%），静态分析工具（SonarQube、CheckStyle）不通过不让合并。
③ **定规范**：统一代码风格、命名规范、错误处理模式，减少因风格差异导致的理解成本。
④ **结对编程 / 技术分享**：对复杂模块，两人一起设计，减少单点理解偏差。
⑤ **对 AI 生成代码额外关注**：review 时重点检查 AI 生成的部分，确认提交者确实理解了代码。

1. 一个 2G 的内存条，while(true){ malloc() }，一次 malloc() 消耗 512M，能做多少次？

答：这道题考察对虚拟内存的理解，答案不是简单的 2G ÷ 512M = 4 次。

**关键点：malloc() 分配的是虚拟地址空间，不是物理内存。**

- 在 64 位 Linux 系统上，进程的虚拟地址空间高达 128TB，malloc() 只是在虚拟地址空间里"划出"一块范围，并不立即消耗物理内存。
- 只有当程序真正**访问**（读写）这块内存时，才会触发缺页中断，OS 才分配物理内存页。
- Linux 默认开启**内存 overcommit**（`/proc/sys/vm/overcommit_memory = 1`），允许分配远超物理内存的虚拟地址，malloc 不会因为物理内存不足而返回 NULL。

**结论：**

- 如果只 malloc 不访问：理论上可以分配非常多次（直到虚拟地址空间耗尽，64位下几乎不可能）。
- 如果 malloc 后立刻访问（memset 等）：分配约 3~4 次后（2GB RAM 扣掉 OS 占用约几百 MB），物理内存耗尽，触发 OOM Killer，进程被强制杀死，而不是 malloc 返回 NULL。
- 如果有 swap：可以多撑几次，直到 swap 也耗尽。

**面试标准答：** 能做 4 次（理论上 2G/512M=4），但实际因为 OS 本身占用内存，可能只能成功 3 次；并且在 Linux 默认配置下，malloc 本身不会失败，而是在访问内存时触发 OOM。

1. malloc() 分配的是什么样的内存？堆内存有哪几种？

答：

**malloc() 分配的是堆上的虚拟内存**，具体行为分两种策略（以 glibc ptmalloc 为例）：

- **小块分配（< 128KB）**：通过 `brk()`/`sbrk()` 扩展堆段，在堆段内用内存池管理（按 chunk 大小分 fast bin / small bin / large bin / unsorted bin），复用已释放的内存块，减少系统调用。
- **大块分配（≥ 128KB）**：直接调用 `mmap()` 申请匿名内存映射，释放时直接 `munmap()` 归还 OS，不进内存池。

**堆内存的几种类型（从管理角度）：**


| 类型                  | 说明                                                   |
| ------------------- | ---------------------------------------------------- |
| 进程堆段（heap segment）  | 由 brk/sbrk 管理，是传统意义上的堆，向高地址增长                        |
| mmap 匿名映射           | 大块 malloc 或直接 mmap(NULL,...,MAP_ANONYMOUS) 得到，不与文件关联 |
| 线程私有堆（thread arena） | 多线程下 ptmalloc 为每个线程维护独立的 arena，减少锁竞争                 |
| 内存池（应用层）            | 业务自己实现的对象池，从系统堆上预申请一大块后自行管理，减少 malloc/free 的系统调用开销   |

1. 现在让你设计一个匹配算法，一个玩家角色有攻击值和防御值两个数据，使用什么样的数据结构和算法去匹配？

答：本质是在二维空间（攻击值 x，防御值 y）中找"最相近"的玩家，属于**最近邻搜索**问题。

**方案一：综合评分 + 有序结构（简单实用）**

- 将攻击值和防御值加权合并为一个综合战力分：`score = w1 * atk + w2 * def`（权重可调）
- 用一个**有序集合**（Java TreeMap / Redis ZSET）维护等待匹配的玩家列表，按 score 排序
- 新玩家加入时，用二分查找找到 score 最接近的玩家，完成匹配
- 时间复杂度：O(log n)
- 缺点：把二维压缩成一维会损失精度，攻击极高但防御极低的玩家可能被错误匹配

**方案二：K-D Tree（精确二维匹配）**

- 构建二维 K-D Tree，节点按 (atk, def) 存储
- 新玩家加入时，在 K-D Tree 中做最近邻搜索，找欧氏距离最近的玩家
- 时间复杂度：构建 O(n log n)，查询平均 O(log n)
- 适合对匹配精度要求高的场景（如竞技对战）

**方案三：分段桶（工程常用）**

- 将攻击值和防御值各自分段（如每 100 为一个区间），玩家进入对应的二维桶
- 优先在同一桶内匹配，无人时扩展到相邻桶
- 工程实现简单，适合大规模在线游戏，延迟可控

**推荐回答思路：**

> 如果追求精度，用 K-D Tree 做二维最近邻匹配；如果追求工程简单、高并发，用综合评分 + ZSET，Redis 原生支持 ZRANGEBYSCORE，天然支持分布式扩展。
